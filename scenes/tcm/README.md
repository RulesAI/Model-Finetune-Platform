## 一、场景定位

为**中医药垂类大模型**提供端到端微调能力 —— 让 GLM-4-9B / Qwen2.5-14B 掌握中医药专业语言、辨证思维、方剂配伍逻辑，输出可被下游业务系统消费的模型 API。

### 服务方向（基于数据清单 9 大类推测）

| 方向 | 数据支撑 | 下游业务场景 |
|------|---------|------------|
| 中医知识体系（夯实底层）| 古籍 50 部 + 期刊 10 万篇 | 医学教育 / 学术研究 |
| 临床诊疗辅助 | 诊疗指南 500 份 + 医案 1 万例 | 中医临床决策辅助 |
| 中药/方剂咨询 | 中药 1 万种 + 方剂 10 万首 | 药企 / 中药房 / 药品咨询 |
| 中医外治科普 | 操作规范 500 套 | 针灸推拿门店 / 大众科普 |
| 康养养生服务 | 养生方案 1 万套 + 药膳 5000 个 | C 端健康咨询 / 体质调理 |
| 中成药用药指导 | 中成药 5000 种 + 说明书 5000 份 | 基层用药咨询 / 药房辅助 |

### 客户群体（潜在）

- 中医药大学 / 中医医院（教学 / 临床决策）
- 中医药企业（药品研发 / 商品咨询）
- 健康管理 / 养生 App（C 端服务）
- 政府机构（国家中医药管理局 / 卫健委标准化项目）

## 三、v3 工具集是否满足 TCM 场景？（核心评估）

### 3.1 综合评分：**约 60% 满足**

工具集大盘可用，但有 **6 个差距**需要扩展。详见下表。

### 3.2 6 个差距分析（G1-G6）

#### G1 关键缺口：CPT 流水线缺失 🔴

| 维度 | 现状 | 需求 | 行动 |
|------|------|------|------|
| v3 流水线 | 5 阶段（数据 → SFT → DPO → 评估 → 部署），跳过 CPT | TCM 数据量 8-10 亿 token，**满足 CPT 阈值**（行业典型 10 亿+ token）| 补 Phase 0 CPT |
| ms-swift 工具本身 | ✅ **已支持 CPT**（官方："CPT/SFT/DPO/GRPO"）| - | 工具够用，**只需补流水线设计** |

**为什么 TCM 必须做 CPT 而 HR 不需要**：
- HR 140 万 token → CPT 几乎学不到东西（差 1000× 阈值），直接 SFT 即可
- TCM 8 亿 token → CPT 可以注入"古文中医知识体系 + 方剂配伍规律 + 辨证思维链路"，这些 SFT 学不来

**v4 必做**：新增 Phase 0 CPT 阶段（详见 `docs/01-训练方案与时间计划-v4.md`）

#### G2 重要缺口：数据处理需并行化 🟡

| 工具 | 现状 | 问题 |
|------|------|------|
| MarkItDown | 单进程文档解析 | 10 万篇期刊论文 × 10 秒/篇 = ~12 天单机不可接受 |
| Distilabel | 单机数据合成 | 100 万 + 条目数据增广，单机周级别 |

**v4 建议**：
- MarkItDown 并行化（Python multiprocessing + 24 进程，3-5 天可完成）
- Distilabel 走 Argilla 分布式集群（Kubernetes）或拆分多机
- 或：放弃完整数据，**采样训练**（先 1 亿 token CPT 验证）

#### G3 风险点：Unsloth 多卡不稳定 🟡

继承 F1 校正：TCM 数据量需多卡训练，**不要用 Unsloth backend**，退回 ms-swift 原生 + DeepSpeed ZeRO。

#### G4 重要缺口：评估缺中医药 benchmark 🟡

| 维度 | 现状 | 需求 |
|------|------|------|
| v3 评估 | C-Eval / CMMLU / QualBench（HR 资格类）/ DeepEval | 缺中医药专门 benchmark |

**v4 建议**：
- 调研 TCM 领域是否有已发布 benchmark（如 TCM-Bench / MedBench 中医部分）
- 如无，**自建**：用国家中医执业医师资格考试题（公开题库）建 500-1000 题评估集
- 路径：`scenes/tcm/eval_set/tcm_doctor_exam.jsonl`

#### G5 未知风险：多模态需求 ⚠️

| 数据类型 | 是否需多模态 |
|---------|------------|
| 古籍文献 | 否（纯文本）|
| 期刊论文 | 论文中**含图表**，可能需 OCR + 图理解 |
| 中药材 | **大概率需要**（药材图谱、识别）|
| 经络穴位 | **大概率需要**（经络图）|
| 舌诊面诊 | **必须**（视觉诊断核心）|

**v4 决策**：
- v1 训练**仅文本**（用 GLM-4-9B-Chat-1M），先把 CPT 流水线跑通
- 如果业务方要求"看图识药材 / 舌诊辅助"，**v2 切多模态基座**（GLM-4.5V / Qwen3-VL）—— 不在本场景 v1 范围

#### G6 资源升级：算力 + 时间 10-20× 🟡

| 维度 | HR 场景 | TCM 场景 | 倍数 |
|------|---------|---------|------|
| 训练时间 | 5 周 | **2-4 个月** | 8-16× |
| GPU 需求 | 单卡 A100 | **8-16 卡 H100**（CPT 阶段） | 8-16× |
| 算力成本 | ¥10000-15000 | **¥10-30 万** | 10-20× |
| 数据预处理时间 | 1-2 周 | **3-6 周** | 3× |
| 团队投入 | 1 算法 + 1 HR 专家 | **2-3 算法 + 2-3 中医专家** | 2-3× |

**v4 建议**：业务方启动前必须确认算力和预算，否则项目无法启动。

### 3.3 修订建议汇总（v4 → 工具集扩展）

| # | 缺口 | 紧迫度 | v4 应做 |
|---|------|--------|---------|
| **G1** | 缺 CPT 流水线 | 🔴 必须 | 新增 Phase 0 CPT（落到 docs/01-v4 + configs/ms_swift/cpt 模板）|
| **G2** | 数据处理并行化 | 🟡 重要 | 增加 `scripts/data_prep/parallel_*.py`（multiprocessing 包装）|
| **G3** | Unsloth 多卡风险 | 🟡 重要 | 沿用 F1：TCM 场景强制禁用 Unsloth，用 DeepSpeed |
| **G4** | TCM benchmark | 🟡 重要 | 调研 + 自建（中医执业医师考试题）|
| **G5** | 多模态扩展 | 🟢 v2 议题 | v1 仅文本；v2 视业务需求评估多模态 |
| **G6** | 算力成本 10-20× | 🔴 业务前置 | 启动前业务方确认预算 |

## 八、潜在协同与挑战

### 8.1 与既有项目协同

- **HR Knowledge Nexus**：架构复用（垂类知识 LLM）
- **GraphForge**：图谱建模（中医药本体是天然的图）
- **未来 manufacturing**：制造业 SOP 与中医操作规范的处理方法可借鉴

### 8.2 中医药领域独特挑战

| 挑战 | 应对方向 |
|------|---------|
| 古今术语对齐（"消渴" = "糖尿病"）| Phase 0 CPT 时混合训练 / 标准术语表 prompt 引导 |
| 中医理论无法用现代医学逻辑解释 | system_prompt 严格约束模型不做"中西医对比错误论断" |
| 错误用药风险高 | DPO 阶段强化"模糊不答 / 推荐就医" |
| 国家中医药管理局规范严 | Phase 0 必须注入诊疗指南，避免输出违规建议 |
