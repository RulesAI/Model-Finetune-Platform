## 二、关键判断（必须先校正认知）

### 判断 1：**这个数据量不适合做 CPT（继续预训练）**

| 阶段 | 行业典型数据量 | 我们现有的 | 差距 |
|------|--------------|----------|------|
| **CPT（继续预训练）** | 10 亿 - 100 亿 token | 140 万 token | **差 1000× 以上** |
| **SFT（监督微调）** | 3k - 10 万条 instruction | 158 篇文档可衍生 5k-15k 条 | **正好匹配** |
| **DPO（偏好对齐）** | 200 - 2000 偏好对 | 可从专家审稿构造 | **充足** |

**结论**：直接做 CPT 等于把 5 滴水倒进游泳池——模型学不到任何东西，还可能过拟合到偏激分布。**应该跳过 CPT，从 SFT 起步，配合 RAG 做知识精确性兜底**。

> 这也是上一轮"语料学习两轮"汇报最大的疑点之一：如果用 140 万 token 跑 CPT，loss 怎么看都不会收敛到健康状态。

### 判断 2：**RAG 必须配套，不能只靠微调**

政策文件需要**精确引用条款编号、生效日期、数字阈值**，微调模型在这些"事实细节"上会幻觉。HR 应用必走**微调（学风格 / 学结构 / 学推理）+ RAG（保事实精度）**双轨。

## 三、GLM 开源模型版本推荐

### 推荐方案：**GLM-4-9B-Chat-1M**（首选）

| 维度 | GLM-4-9B-Chat-1M |
|------|------------------|
| 参数量 | 9B（90 亿） |
| 上下文 | **100 万 token** ← 单文档可整篇喂入，国企政策动辄几万字非常受用 |
| 开源协议 | 智谱开源协议（商用需登记） |
| 硬件门槛 | LoRA：单卡 A100 80G 或 2×4090 24G<br>全参 SFT：4-8 卡 A100 80G |
| HR 领域适配 | 中文场景原生强，CMMLU / C-Eval 同尺寸 SOTA |

**为什么不选其他版本**：

| 候选 | 否决理由 |
|------|---------|
| ❌ GLM-5（745B） | 部署成本极高，2 台 N 卡机器跑不动；74B 也只够推理不够全参 SFT |
| ❌ GLM-4.5（355B） | 同上，过大 |
| ❌ GLM-4-32B | 适合算力充足场景；但 HR 这种窄领域，9B 已够，32B 性价比反不如 9B |
| ❌ ChatGLM3-6B | 已停止主线维护，工具链过时 |
| ⚠️ GLM-4-9B-Chat（标准版） | 上下文 128K，比 1M 短，处理长政策文档需切片，**次选** |

### 升级路径

如果第一轮 9B 效果不达标且算力充足 → 升级到 **GLM-4-32B**（约 4× 显存需求）。

## 四、性价比最高的任务清单（4 阶段流水线）

按 ROI 排序，前两阶段必做、第 3 阶段强烈推荐、第 4 阶段可选。

### Phase 1：数据工程（性价比 ★★★★★，最高 ROI）

**任务**：把 158 篇 markdown 转成 SFT 训练数据。

| 子任务 | 产出 | 工作量 |
|--------|------|--------|
| 1.1 文档清洗去重 + 脱敏二次校验 | 干净的 markdown 库 | 2 天 |
| 1.2 构造 3 类 instruction 数据 | 见下表 | 5-7 天 |
| 1.3 GPT-4 / Claude 蒸馏增广 | 5k → 15k 条数据 | 2-3 天 |
| 1.4 切 train / dev / test = 8:1:1 | 训练就绪数据集 | 0.5 天 |

**3 类指令模板**（HR 场景最高频）：

| 类型 | 占比 | 样例 |
|------|-----|------|
| **问答型 QA** | 50% | Q: "国企工资总额备案制和核准制有什么区别？" A: 从《39号令》提取 |
| **方案生成** | 30% | Q: "为某省属能源国企设计中长期激励方案" A: 参考央企案例输出 |
| **抽取与对比** | 20% | Q: "对比《39号令》与《工效挂钩办法》在工资总额预算上的差异" A: 结构化输出 |

> 数据工程占总工作量 40%，是微调成败的关键。低质量 1 万条 < 高质量 3000 条。

### Phase 2：SFT 监督微调（性价比 ★★★★★）

**任务**：用 LoRA 在 GLM-4-9B-Chat-1M 上做 SFT。

| 配置项 | 推荐值 |
|--------|--------|
| 训练方法 | **LoRA**（r=16，alpha=32，target=all linear） |
| 学习率 | 2e-4（LoRA 比全参高一档） |
| Epoch | 2-3（小数据集别贪心） |
| batch size | 32（梯度累积） |
| 最大序列长度 | 8192（覆盖大部分政策段落） |
| 数据回放 | 混入 5% 通用 instruction 数据（防止 HR 化过深） |

**预期产出**：第一个可用 checkpoint，相比 zero-shot GLM-4-9B 在 HR benchmark 上提升 15-25%。

### Phase 3：DPO 偏好对齐（性价比 ★★★★）

**任务**：让模型在合规性、引用准确性、表述严谨度上对齐 HR 专家。

| 配置项 | 推荐值 |
|--------|--------|
| 偏好对数量 | 500-1000 对 |
| chosen 来源 | HR 专家审定的"好答案" |
| rejected 来源 | SFT 模型次优输出 / 故意构造的合规违规版 |
| 训练方法 | DPO（不要 PPO，DPO 更稳） |
| 学习率 | 5e-7（DPO 极低） |
| Epoch | 1 |

**预期产出**：在风格 / 合规 / 严谨度三个维度，相比纯 SFT win-rate ≥ 60%。

### Phase 4：RAG 知识库搭建（性价比 ★★★★★，与微调并行）

**任务**：把 158 篇文档建成可检索的知识库。

| 配置项 | 推荐值 |
|--------|--------|
| 向量库 | ChromaDB / Milvus / pgvector |
| Embedding 模型 | BGE-large-zh-v1.5（中文最强 embedding 之一） |
| 切片策略 | 按章节切，每片 500-1000 字，10% overlap |
| 检索器 | Hybrid（向量 + BM25）+ rerank |

**为什么必做**：政策条款的精确数字（如"工资总额增幅不超过 XX%"）微调记不住，必须 RAG 检索原文兜底。

## 六、评估方案（每周必做）

### 6.1 自动评估（每个 checkpoint）

| 指标 | 工具 | 通过线 |
|------|------|--------|
| **训练 loss** | tensorboard | 收敛、不再下降 |
| **验证集 PPL** | 自跑脚本 | 比 zero-shot 降 30%+ |
| **ROUGE-L**（生成任务） | rouge-chinese 库 | ≥ 0.45 |
| **Exact Match**（抽取任务） | 自跑脚本 | ≥ 0.75 |

### 6.2 通用能力回归（每轮训练后必跑）

| benchmark | 抽样量 | 通过线 |
|-----------|-------|--------|
| **C-Eval** | 200 题 | 比 base 模型掉点 < 3% |
| **CMMLU** | 200 题 | 比 base 模型掉点 < 3% |
| **MMLU**（英文） | 100 题 | 比 base 模型掉点 < 5% |

> 这是最容易翻车的环节：HR 指标涨了 20 点但通用能力掉 15 点 = 失败。

### 6.3 人工评估（每个里程碑）

| 评估维度 | 评分 | 通过线 |
|---------|------|--------|
| 事实准确性 | 1-5 | ≥ 4.0 |
| HR 合规性 | 1-5 | ≥ 4.5（红线指标） |
| 语言流畅度 | 1-5 | ≥ 4.0 |
| 有用性 | 1-5 | ≥ 4.0 |

每个 checkpoint 抽 100 条样本，由 1-2 名 HR 专家盲评。

## 九、与"已有进度"的对接建议

如果团队已经按"两轮语料学习"的思路在跑 CPT：

1. **立即停止当前 CPT**（数据量不够，跑了也白跑）
2. **保留预训练 checkpoint 作为对比基线**，不要丢
3. **从 GLM-4-9B-Chat-1M 原始权重重新起步**，按本方案 Phase 1 → 4 重做
4. **6 周后**与团队当前路线产出的模型做 A/B 对比，用数据说话
