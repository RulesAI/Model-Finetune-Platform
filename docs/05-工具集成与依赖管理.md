> **创建**：2026-05-25

## 二、7 种集成方式速查

| 方式 | 适用情景 | 优势 | 劣势 |
|------|---------|------|------|
| **① pip install** | 库形态工具（distilabel / markitdown / presidio）| 升级简单、版本锁定 | 需联网下载 |
| **② pip install + CLI 调用** | 既是库又有 CLI（ms-swift / lmdeploy / opencompass）| subprocess 解耦 | - |
| **③ Docker 部署** | 服务型工具（Argilla / Prometheus / Grafana）| 隔离环境、易部署 | 启动慢 |
| **④ git submodule / vendoring** | 需修改源码或离线场景 | 完全自主可控 | 升级麻烦、占空间 |
| **⑤ ModelScope / HF Hub 下载** | 模型权重（GLM / Qwen 基座）| 自动管理 cache | 文件大（10-30GB）|
| **⑥ 外部 API 调用** | 教师模型（Claude / GPT-4o）| 零部署成本 | 涉密慎用 |
| **⑦ 离线包 + 内网镜像** | 涉密内网部署 | 满足合规 | 升级流程复杂 |

## 三、5 层依赖管理架构

```
┌─────────────────────────────────────────────────────────────┐
│ Layer 1: PyPI 依赖（占 80%）                                 │
│ - requirements.txt 锁版本                                    │
│ - pyproject.toml 元数据 + 可选 extras                        │
└─────────────────────────────────────────────────────────────┘
                            +
┌─────────────────────────────────────────────────────────────┐
│ Layer 2: Docker Compose（服务型，占 10%）                    │
│ - argilla / prometheus / grafana / litellm-prod             │
└─────────────────────────────────────────────────────────────┘
                            +
┌─────────────────────────────────────────────────────────────┐
│ Layer 3: 模型权重（占 5%）                                   │
│ - ModelScope / HF 下载到 cache 目录                          │
└─────────────────────────────────────────────────────────────┘
                            +
┌─────────────────────────────────────────────────────────────┐
│ Layer 4: 外部数据集（占 3%）                                 │
│ - benchmark 数据（C-Eval / CMMLU / QualBench）              │
└─────────────────────────────────────────────────────────────┘
                            +
┌─────────────────────────────────────────────────────────────┐
│ Layer 5: 自写代码（占 2%）⭐ 本项目真正的工程产出             │
│ - scripts/ / backend/ / frontend/ / configs/ / scenes/      │
└─────────────────────────────────────────────────────────────┘
```

**关键洞察**：本项目自写代码量预期仅占 2%，剩 98% 是引用开源工具 + 模型权重。这是**整合层项目**的本质。

## 五、Layer 1：PyPI 依赖管理

### 5.1 requirements.txt 设计原则

- **版本锁定**：每个依赖锁到精确版本（`==`），不是 `>=`
- **分组组织**：按 Phase 分组，注释说明用途
- **每月评审**：每月 review 一次有无 critical bug fix 需升级

### 5.2 requirements.txt 骨架（启动时落地）

```text
# ============================================
# Model-Finetune-Platform v4 依赖清单
# 锁定日期：2026-05-25
# 版本号占位，启动时填精确版本
# ============================================

# --- Phase 0 + 2 + 3 训练 ---
ms-swift==4.2.0
deepspeed==X.X.X
unsloth==X.X.X            # F1: 仅小数据集场景启用
swanlab==X.X.X
tensorboard==X.X.X
# llamafactory（备选，按需安装）

# --- Phase 1 数据工程 ---
markitdown[all]==X.X.X
distilabel==X.X.X
presidio-analyzer==X.X.X
presidio-anonymizer==X.X.X
anthropic==X.X.X           # 教师模型 API（涉密场景注释掉）
openai==X.X.X              # 备选教师模型 API

# --- Phase 4 评估 ---
opencompass==X.X.X
deepeval==X.X.X
rouge-chinese==X.X.X
# argilla-server 走 Docker，不在此

# --- Phase 5 部署 ---
lmdeploy==X.X.X
vllm==X.X.X
litellm==X.X.X

# --- 自建后端 / 前端工具链不在此 ---
# Python 后端依赖见 backend/requirements.txt
# 前端依赖见 frontend/package.json
```

### 5.3 pyproject.toml 可选 extras 设计

```toml
[project]
name = "model-finetune-platform"
version = "0.1.0"

[project.optional-dependencies]
hr = [
    # HR 场景特定（不含 CPT 大依赖）
]
tcm = [
    # TCM 场景特定（含 DeepSpeed、CPT 工具）
    "deepspeed==X.X.X",
]
dev = [
    "pytest==X.X.X",
    "black==X.X.X",
    "ruff==X.X.X",
]
```

**用法**：
```bash
pip install -e .              # 基础依赖
pip install -e .[hr]          # + HR 场景
pip install -e .[tcm]         # + TCM 场景（含 CPT）
pip install -e .[hr,tcm,dev]  # 全部
```

### 5.4 版本锁定原则

| 依赖类型 | 锁定粒度 | 升级周期 |
|---------|---------|---------|
| 训练核心（ms-swift / DeepSpeed）| 精确版本 `==` | 季度评审 |
| 评估工具（OpenCompass / DeepEval）| 精确版本 `==` | 季度评审 |
| 推理引擎（LMDeploy / vLLM）| 精确版本 `==` | 季度评审 |
| 工具型库（MarkItDown / Presidio）| 精确版本 `==` | 半年评审 |
| 外部 API SDK | `~=X.Y.0` 允许 patch 升级 | 自动 |

## 六、Layer 2：Docker 编排

### 6.1 docker-compose.yml 服务清单

```yaml
# docker-compose.yml（启动时落地骨架）
version: '3.8'

services:
  argilla:
    image: argilla/argilla-quickstart:latest
    ports:
      - "6900:6900"
    volumes:
      - argilla-data:/var/lib/argilla

  prometheus:
    image: prom/prometheus:latest
    ports:
      - "9090:9090"
    volumes:
      - ./configs/monitoring/prometheus.yml:/etc/prometheus/prometheus.yml

  grafana:
    image: grafana/grafana:latest
    ports:
      - "3007:3000"   # 避免与项目前端 3006 冲突
    depends_on:
      - prometheus

  litellm:                                    # 生产网关
    image: ghcr.io/berriai/litellm:main-latest
    ports:
      - "4000:4000"
    environment:
      - LITELLM_MASTER_KEY=${LITELLM_KEY}

volumes:
  argilla-data:
```

### 6.2 Dockerfile 多 stage 设计

| 镜像 | 用途 | base | 大小估算 |
|------|------|------|---------|
| **Dockerfile.training** | CPT/SFT/DPO 训练 | nvidia/cuda:12.4-devel-ubuntu22.04 | ~15 GB |
| **Dockerfile.inference** | LMDeploy 推理 | nvidia/cuda:12.4-runtime-ubuntu22.04 | ~8 GB |
| **Dockerfile.dataprep** | Phase 1 数据处理（无 GPU）| python:3.10-slim | ~2 GB |
| **Dockerfile.backend** | FastAPI 后端 | python:3.10-slim | ~1 GB |

### 6.3 训练镜像示例骨架

```dockerfile
# Dockerfile.training
FROM nvidia/cuda:12.4-devel-ubuntu22.04

# 系统依赖
RUN apt-get update && apt-get install -y \
    python3.10 python3-pip git curl \
    && rm -rf /var/lib/apt/lists/*

# Python 依赖（先装 PyTorch，再装训练栈）
RUN pip install torch==2.4.0 --index-url https://download.pytorch.org/whl/cu124
COPY requirements.txt /tmp/
RUN pip install -r /tmp/requirements.txt

WORKDIR /workspace
# 训练时挂载项目目录 + cache + checkpoints
```

## 七、Layer 3：模型权重管理

### 7.1 ModelScope CLI 下载流程

```bash
# 一次性下载基座模型（约 18 GB）
pip install modelscope
modelscope download --model ZhipuAI/glm-4-9b-chat-1m --local_dir ~/.cache/modelscope/hub/ZhipuAI/glm-4-9b-chat-1m

# Qwen2.5-14B（约 28 GB）

```

### 7.2 项目内 `models/` 软链方案

```bash
# 项目内不存模型权重，仅建软链方便引用
ln -s ~/.cache/modelscope/hub models/modelscope
ln -s ~/.cache/huggingface/hub models/huggingface
```

`models/` 整目录加入 `.gitignore`，仅保留 `models/README.md` 说明下载流程。

### 7.3 涉密内网 Cache 重定向

```bash
# 涉密场景：模型权重通过 U 盘传到指定路径
mkdir -p /data/models/
cp -r /usb/glm-4-9b-chat-1m /data/models/

# 通过环境变量重定向 cache
export MODELSCOPE_CACHE=/data/models/
export HF_HOME=/data/models/
```

## 十一、为什么默认不 vendoring（设计决策）

### 11.1 对比

| 因素 | pip 集成 ✅ 推荐 | vendoring ❌ 不推荐 |
|------|----------------|-------------------|
| 项目体积 | venv 单独，几百 MB | 加几 GB 进 git 仓库 |
| 升级简单度 | `pip upgrade` | 手动 git rebase / cherry-pick |
| 上游 bug 修复 | 等下个 release | 自己接管维护 |
| 团队工程师上手 | 与官方文档一致 | 要看本项目改动 |
| Bug 反馈通道 | 上游 GitHub Issue | 内部沟通 |
| 与开源社区对齐 | ✅ | 易过时 |

### 11.2 例外（仅以下场景才 vendoring）

写进 `third_party/README.md`：

1. 上游 bug 阻断 + 短期不会修
2. 涉密内网完全无外网（先批量下载 wheel + Docker image 即可，**不需要 vendoring 源码**）
3. 需要深度 monkey patch 改写核心逻辑（罕见，应优先尝试 PR 给上游）

**默认 `third_party/` 为空目录**，仅有 README 说明使用规则。

## 十四、关键风险与对策

| 风险 | 规避策略 |
|------|---------|
| 依赖版本冲突（PyTorch / CUDA / 工具）| pyproject.toml + 精确版本 + CI 自动测试 |
| Docker image 体积过大 | 多 stage build + alpine 基础镜像（适用部分服务）|
| 模型权重占空间 | 项目内仅软链，实际存 `/data/models/` |
| 涉密客户无 Docker | 提供 `pip install + supervisord` 备选方案 |
| pip 镜像在客户内网不通 | 离线 wheel 包 + `pip install --no-index --find-links wheels/` |
| 升级破坏既有功能 | requirements.txt 锁版本 + CI 全量回归 |

## 十五、一句话摘要

**本项目采用 5 层依赖管理架构（PyPI 80% + Docker 10% + ModelScope 5% + 外部数据集 3% + 自写代码 2%），默认不 vendoring 开源工具源码（升级简单 + 与社区对齐），所有第三方工具通过 `pip install` / `docker pull` / `modelscope download` 三种方式动态拉取，自写代码占比仅 2% —— 本质是整合层项目，不是从零造工具；涉密客户走离线包 + 内网镜像变体，待启动时单独落地 `docs/06-涉密内网部署指南.md`。**
