## 三、Phase by Phase 详细评估

### 3.1 Phase 1 数据工程（评分 95%）

| 工具 | 替代品对比 | 结论 |
|------|----------|------|
| **MarkItDown** | vs Unstructured / LlamaParse | ✅ 微软出品，多格式（PDF/docx/xlsx/HTML/图片 OCR）覆盖最全，中文场景实测稳定 |
| **Distilabel** | vs DataDreamer（ACL 2024）/ Augmentoolkit | ✅ Argilla 出品，与 Argilla 标注平台无缝集成；DataDreamer 学术严谨但工程化弱；建议主用 Distilabel，对照参考 DataDreamer |
| **Presidio** | vs DataFog | ⚠️ Presidio 国际成熟但中文 PII 字典需补；DataFog 国内无成熟方案 |

**Phase 1 整体**：高质量选型，仅 Presidio 需小幅中文化定制。

### 3.2 Phase 2+3 训练（评分 92%）

| 工具 | 关键数据 | 结论 |
|------|---------|------|
| **ms-swift** | v4.0=2026-03-03，v4.2=2026-05-07，**仅 2 月内 2 个 minor 版本** | ✅ 极活跃维护；600+ LLM + 400+ MLLM；集成 vLLM/SGLang/LMDeploy 推理（Phase 5 无缝衔接）|
| **LLaMA-Factory** | 13K+ stars（2024 数据），ACL 2024 论文 | ✅ 国际事实标准；生产环境需补 MLOps（本项目自建 backend 已覆盖）|
| **Unsloth** | Red Hat 官方背书；**多卡仍升级中** | ⚠️ 单卡 + LoRA + 小数据集 → production-ready；多卡场景 → 退回 ms-swift 原生 backend |
| **SwanLab** | 国产 wandb 替代 | ✅ 国内主流，**翻墙问题规避**；国际生态弱（不打算上 wandb 云端）|

**Phase 2+3 整体**：核心选型对，Unsloth 启用需场景化约束。

## 四、5 个关键洞察

### 洞察 1：整体大盘正确（92% 准确率）

12 项工具有 10 项完全是行业最佳实践，2 项有可控风险（Unsloth 多卡 / QualBench 新生）。**比 v1 (75%) / v2 (90%) 持续提升**。

### 洞察 2：Unsloth 多卡是 v3 的最大隐性风险

Red Hat 官方文档原话：**"Multi-GPU works but a much better version is coming"**。

- 单卡 LoRA → production-ready
- 多卡 LoRA → 能跑但不稳定，升级中

**对我们的影响**：HR 158 md（140 万 token）小数据集，**LoRA 单卡 A100 完全够** → 风险不会触发。未来制造/供应链场景如数据上千万 token，多卡时切回 ms-swift 原生 backend。

### 洞察 3：QualBench 的局限

EMNLP 2025 论文亮点：
- 17,000+ 题，24 个中国资格考试，6 个垂直领域
- **关键发现**：Qwen2.5 在中文资格考试上**超过 GPT-4o**
- 平均通过率仅 53.98%（揭示行业差距）

**对我们的影响**：
- ✅ 含人力资源管理师类题目，HR 场景适用
- ⚠️ 原论文**未评估 GLM-4 系列**（只评 ChatGLM3 老版本）→ 需自跑 GLM-4 base baseline

### 洞察 4：ms-swift v4 持续升级证明工程实力

- v4.0：2026-03-03
- v4.2：2026-05-07（仅 2 月内出第 2 个 minor 版本）
- 已集成 vLLM / SGLang / **LMDeploy** 三大推理引擎

**深度协同价值**：ms-swift 训练完直接产出 LMDeploy 兼容模型格式，**Phase 2→Phase 5 零适配成本**。这是 v2 时未意识到的"工具栈一致性收益"。

### 洞察 5：Qwen2.5 在 Plan B 位置可能被低估

QualBench EMNLP 2025 论文原文：
> "Qwen2.5 outperforming the more advanced GPT-4o, emphasizing the value of localized domain knowledge"

**含义**：中文垂直领域 Qwen2.5 表现超过 GPT-4o —— 比 GLM-4 在 C-Eval 87.2% 数据更有说服力（真实垂直考试）。

**建议**：Qwen2.5-14B 从"Plan B 兜底"升格为"主选并行候选"，第一轮 SFT 双基座 A/B（多 1-2 天工作量）。

## 五、4 个校正点详解（F1-F4）

### F1：Unsloth 使用约束

**问题**：v3 工具栈把 Unsloth 列为 backend 加速选项，但未注明使用条件。

**校正建议**：

在 `CLAUDE.md §三 技术栈` 加约束行：
```markdown
| Backend | Unsloth（仅单卡 + LoRA + 小数据集场景）| 多卡 / 全参 / 大数据集场景退回 ms-swift 原生 backend |
```

在 `configs/ms_swift/scenes/<scene>/sft.yaml` 默认配置注释：
```yaml
# Unsloth backend 默认启用（HR 场景是单卡 + LoRA + 140 万 token，符合条件）
# 制造 / 供应链场景如数据量 > 1000 万 token，注释掉以下行：
use_unsloth: true
```

### F2：QualBench 前置 GLM-4 baseline

**问题**：QualBench 原论文未评估 GLM-4 系列，直接拿来评估微调后模型，缺少对照基线。

**校正建议**：

`docs/01-训练方案与时间计划-v3.md §四 Phase 4` 加 baseline 步骤：

```markdown
### Phase 4 评估（含 baseline）

**Step 4.0（v4 新增）**：评估前先跑 base 模型 baseline
- 用 GLM-4-9B-Chat-1M 原始权重在 QualBench HR 资格类 100 题跑分
- 输出 baseline 报告到 `logs/eval/<scene>/baseline_qualbench.md`

**Step 4.1**：评估微调后模型
- 对比 baseline，提升 < 10% 视为微调效果不显著
- 提升 ≥ 15% 为合格，≥ 20% 为优秀
```

### F3：Qwen2.5-14B 升格为并行候选

**问题**：QualBench 实证 Qwen2.5 在中文垂直领域超 GPT-4o，目前仅作 Plan B 兜底可能浪费机会。

**校正建议**：

`docs/01-训练方案与时间计划-v3.md §三 模型版本推荐` 调整：

| v3 当前 | v4 建议 |
|--------|--------|
| 主选 GLM-4-9B-Chat-1M | 主选 GLM-4-9B-Chat-1M |
| Plan B Qwen2.5-14B（兜底）| **并行候选** Qwen2.5-14B（第一轮 SFT 双基座 A/B）|

`scripts/training/sft.py` 支持 `--base-model` 参数同时跑两个基座。

W3 SFT 训练增加约 1-2 天工作量，但通过 QualBench A/B 对比能拿到**实测数据决策**，避免凭印象选错基座。

### F4：Presidio 中文 PII 字典前置定制

**问题**：Presidio 国际方案，中文 PII 检测开箱不准。

**校正建议**：

`docs/03-场景扩展指南.md §六 Scene Checklist 启动前` 加一项：

```markdown
- [ ] **Presidio 中文 PII 字典已定制**（提前 0.5 天，复用 HR 场景的）
   - 中文姓名 / 身份证号 / 手机号 / 银行卡号 / 邮箱正则
   - 行业特定：员工编号 / 工号 / 部门代码格式
```

`scenes/hr-salary/` 新增子目录：
```
scenes/hr-salary/
└── pii_rules/
    ├── chinese_common.yaml    # 通用中文 PII（姓名/身份证/手机）
    └── hr_specific.yaml       # HR 特定（员工编号/薪资字段）
```

其他场景复用 `chinese_common.yaml`，仅写各自 `<scene>_specific.yaml`。
