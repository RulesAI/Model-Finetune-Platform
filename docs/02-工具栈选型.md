# GLM 微调工具栈选型方案

> 输出日期：2026-05-25
> 配套文档：`GLM微调方案与训练时间计划.md`
> 目标：明确 6 阶段流水线（数据 / SFT / DPO / RAG / 评估 / 部署）每一步用什么工具、智谱官方是否提供、开源替代方案如何选

---

## 一、智谱官方工具链现状（2026-05 实地核查）

### 1.1 智谱官方提供什么

| 资源 | 仓库 / 渠道 | 覆盖范围 | 生产可用性 |
|------|-----------|---------|----------|
| **GLM-4 微调 demo** | `github.com/zai-org/GLM-4`（原 THUDM/GLM-4） | LoRA / 全参 SFT 基础脚本 + 配置样例 | ⚠️ **demo 级**，需自行扩展 |
| **ChatGLM3 finetune demo** | `github.com/zai-org/ChatGLM3` | P-Tuning v2 + 全参 + LoRA | ❌ 已过时（ChatGLM3 不再主线维护） |
| **智谱 BigModel API 平台** | `bigmodel.cn` | 云端微调 + 部署 | ❌ **不适合 HR 场景**（数据上云、涉密风险） |
| **GLM 技术论文** | arxiv | 训练方法 + benchmark | 📖 仅文档参考 |

### 1.2 智谱官方**没有**提供的能力

| 缺口 | 影响 |
|------|------|
| 数据清洗 / 指令构造 / 蒸馏增广工具 | 占整体工作量 40% 的环节完全裸奔 |
| DPO / RLHF 训练 | 偏好对齐能力缺失 |
| RAG 工具栈 | HR 场景必备（政策数字 / 条款引用）完全没有 |
| 评估 / benchmark 工具 | 无法量化进度，回到上一个问题的"完成两轮"黑盒 |
| 高性能推理引擎 | 仅 transformers 原生推理，吞吐低 5-10× |
| 训练监控 / 实验管理 | 多轮微调无法追溯 |

### 1.3 关键判断

**智谱官方只覆盖训练流水线的 1/6，且只是 demo 级。** 想做生产级 HR 微调，**必须用开源工具拼装**——但好消息是 2026 年中文 LLM 工具链非常成熟，所有缺口都有 SOTA 开源方案。

## 二、6 阶段工具栈推荐（最终选型）

### Phase 1：数据工程

| 子任务 | 推荐工具 | 替代方案 | 选择理由 |
|-------|---------|---------|---------|
| **文档解析**（md / pdf / docx） | **MarkItDown**（微软） | Unstructured / LlamaParse | 我们 158 份已是 md，但残余 pdf 转换走 MarkItDown 最稳 |
| **文档切片** | **LlamaIndex 的 SentenceSplitter** | LangChain TextSplitter | LlamaIndex 切片对中文更友好 |
| **去重 / 清洗** | **Data-Prep-Kit (IBM)** + 自写脚本 | MinHash / SimHash | 158 份小数据集，自写脚本足够 |
| **指令构造** | **Distilabel**（Argilla 出品） | Augmentoolkit / 自写脚本 | Self-Instruct + EvolInstruct 工业级实现 |
| **蒸馏增广**（用大模型扩样） | **Distilabel + Claude-4.6 / GPT-4o API** | 直接调智谱 GLM-4.6 API | Claude 中文蒸馏质量略优，GLM-4.6 成本更低 |
| **脱敏二次校验** | **Presidio**（微软）+ 自写正则 | DataFog | Presidio 中文支持需补字典，但开源最成熟 |
| **数据集格式转换** | LLaMA-Factory 内置的 dataset converter | - | 与下一阶段无缝衔接 |

**Phase 1 总工具数**：5-7 个，全部开源免费（仅蒸馏 API 付费约 ¥3000-5000）。

### Phase 2：SFT 监督微调（最核心）

#### 三大候选框架对比（2026-05 实测数据）

| 维度 | **LLaMA-Factory** ⭐ 推荐 | **Unsloth** | **Axolotl** |
|------|-------------------------|-------------|-------------|
| 训练速度（A100 8B QLoRA 基准） | 3.4h（接 Unsloth 后端） | **3.2h（最快）** | 5.8h |
| GLM-4 / GLM-4.5 支持 | ✅ 一等公民 | ✅ 支持 | ✅ 支持 |
| Web UI | ✅ LlamaBoard | ❌ 纯代码 | ❌ 纯代码 |
| YAML 配置驱动 | ✅ | 部分 | ✅ |
| DPO / KTO / ORPO | ✅ 全支持 | ⚠️ 部分 | ✅ |
| 多机多卡 / DeepSpeed | ✅ | ⚠️ 限制多 | ✅ 最强 |
| 中文文档 / 社区 | ✅ 国人维护，**最佳** | ⚠️ 英文为主 | ⚠️ 英文为主 |
| 学习曲线 | 平缓 | 平缓 | 陡峭 |

**推荐：LLaMA-Factory**。理由：
1. GLM 系列原生一等支持（毕竟智谱模型 + 国内团队维护）
2. 一个框架覆盖 SFT + DPO 两个阶段
3. 中文文档最全，团队上手快
4. 配 `--use_unsloth` 后端速度只比裸 Unsloth 慢 6%

**不推荐 Unsloth 单独使用**的原因：DPO 等高阶能力支持不完整，且不适合多机多卡。

#### 配套工具

| 子任务 | 工具 |
|-------|------|
| 训练监控 | **TensorBoard**（基础）+ **Weights & Biases**（推荐，可云可本地）+ **SwanLab**（国产 wandb 替代，无墙） |
| 实验管理 | **SwanLab** 或 **MLflow** |
| Checkpoint 管理 | LLaMA-Factory 内置 + 手动按时间 / 指标命名 |

### Phase 3：DPO 偏好对齐

| 子任务 | 工具 | 替代 |
|-------|------|------|
| **偏好对构造** | **Distilabel**（UltraFeedback 流水线） | 自写脚本 + 专家审 |
| **DPO 训练** | **LLaMA-Factory**（推荐，与 Phase 2 同框架） | **TRL**（HuggingFace 官方，更底层） |
| **偏好评估** | **RewardBench**（开源 benchmark） | 自建测试集 |

**强烈建议沿用 LLaMA-Factory**，避免再切换工具栈。

### Phase 4：RAG 知识库（与训练并行）

#### 框架选型

| 维度 | **RAGFlow** ⭐ 推荐 | **LlamaIndex** | **LangChain** |
|------|-------------------|----------------|---------------|
| 文档解析（复杂 PDF） | ✅ **最强**（DeepDoc OCR） | ⚠️ 中等 | ⚠️ 一般 |
| 中文优化 | ✅ 国人团队 | ⚠️ 通用 | ⚠️ 通用 |
| 开箱即用 UI | ✅ 完整 Web UI | ❌ | ❌ |
| 知识库管理 | ✅ 多租户 | 需自建 | 需自建 |
| 引用追溯 | ✅ 强 | ✅ 强 | ⚠️ 中 |
| 灵活定制 | ⚠️ 中（包得较死） | ✅ 最灵活 | ✅ 最灵活 |

**推荐：RAGFlow**。理由：
1. HR 政策文档多为 PDF 表格，DeepDoc 解析准确率显著高于其他方案
2. 自带 Web 管理界面，HR 业务方可直接维护知识库
3. 国产工具，中文支持 + 文档社区最完善
4. 政策条款的精确引用追溯能力强（关键场景）

**备选方案**：如果团队已有 LlamaIndex 技术栈，沿用 LlamaIndex + 自建 UI 也可。

#### Embedding 与 Rerank 模型

| 用途 | 推荐 | 备选 |
|------|------|------|
| **Embedding** | **BGE-M3**（智源 BAAI）—— 多语言 + 多模态 + 长上下文 | BGE-large-zh-v1.5 / Qwen3-Embedding |
| **Rerank** | **BGE-reranker-v2-m3** | Qwen3-Reranker |

#### 向量库

| 选项 | 适用场景 |
|------|---------|
| **ChromaDB** | 小规模 < 10 万 chunk，本地 dev 首选 |
| **Milvus** | 中大规模生产，需要分布式 |
| **pgvector** | 已有 PostgreSQL 基础设施 |

**HR 158 份文档预计切出 3000-8000 chunk**，**ChromaDB 完全够用**，无需复杂方案。

### Phase 5：评估（每个 checkpoint 必跑）

| 评估类型 | 工具 | 备选 | 选择理由 |
|---------|------|------|---------|
| **通用能力 benchmark** | **OpenCompass**（上海 AI Lab） | lm-evaluation-harness | OpenCompass 中文 benchmark 最全（C-Eval / CMMLU / AGIEval 全覆盖） |
| **RAG 系统评估** | **RAGAS** | LlamaIndex Evaluator | RAG 四件套（faithfulness / answer relevancy / context precision / context recall）业界标准 |
| **业务测试用例** | **DeepEval**（pytest 风格） | Promptfoo | DeepEval 像 pytest，可接入 CI/CD |
| **人工评估平台** | **Argilla**（开源） | Label Studio | Argilla 专为 LLM 标注设计，HR 专家友好 |
| **LLM-as-Judge** | DeepEval + Claude-4.6 / GPT-4o | 自写 prompt | 用强模型评弱模型 |

**最小可用组合**：OpenCompass（通用能力回归）+ RAGAS（RAG 评估）+ DeepEval（业务 CI）三件套。

### Phase 6：推理部署（生产上线）

| 维度 | **vLLM** ⭐ 推荐 | **SGLang** | **LMDeploy** |
|------|----------------|-----------|--------------|
| 吞吐（H100，tokens/s） | 12,500 | **16,200** | 16,200 |
| GLM-4 / GLM-5 支持 | ✅ 官方一等 | ✅ 官方一等 | ✅ 官方一等 |
| 生态成熟度 | ✅ **最好** | ⚠️ 较新 | ⚠️ 较新 |
| 多轮对话优化（RadixAttention） | ⚠️ 一般 | ✅ **最强** | ⚠️ 一般 |
| 量化部署（INT4 / FP8） | ✅ | ✅ | ✅ **最强** |
| 文档 / 社区 | ✅ 最丰富 | ✅ 增长快 | ✅ 智谱内部用 |

**推荐：vLLM**。理由：
1. 生态最成熟，HR 系统接入文档最多
2. GLM-4 series 官方支持完善
3. 吞吐与 SGLang 差距仅 23%，对 HR 场景（QPS 不高）完全可接受
4. 与 LangChain / RAGFlow 集成最丝滑

**如果将来追求极致吞吐**（QPS > 100）：升级 SGLang。

#### 部署配套

| 用途 | 工具 |
|------|------|
| 模型量化 | **AutoGPTQ / AutoAWQ**（推理前） / vLLM 内置 INT8 |
| API 网关 | **LiteLLM**（统一 OpenAI 兼容接口）/ **Xinference** |
| 监控 | **Prometheus + Grafana**（vLLM 已内置 metrics 端点） |

## 四、智谱官方 vs 开源工具栈成本对比

| 维度 | 智谱官方方案 | 开源拼装方案 ⭐ |
|------|------------|----------------|
| 完整度 | 1/6 阶段（仅 SFT demo） | 6/6 阶段全覆盖 |
| 数据上云风险 | 高（HR 数据涉密） | 零（全本地） |
| 软件许可成本 | 智谱 API 按 token 计费 | 全免费（仅蒸馏教师模型 ¥3-5k） |
| 学习成本 | 低（官方文档） | 中（5-7 个工具） |
| 灵活性 | 低（黑盒） | 高（每个组件可换） |
| 长期可维护性 | 锁定智谱生态 | 行业主流栈，团队迁移容易 |
| 性能上限 | 受智谱 API 限制 | 可优化到硬件极限 |

**结论**：开源拼装是唯一可行路径。

## 六、风险与避坑清单

| 风险 | 规避策略 |
|------|---------|
| LLaMA-Factory 配置文件踩坑 | 直接复制 `examples/train_lora/glm4_lora_sft.yaml` 起步 |
| Unsloth 与 GLM-4 不完全兼容 | 第一轮先用 LLaMA-Factory 纯 PyTorch backend 跑通，再加 Unsloth 加速 |
| RAGFlow 解析 PDF 慢 | 我们 158 份已是 md，跳过 RAGFlow 的 OCR，直接 chunk |
| BGE-M3 中文短文本检索召回不稳定 | 配合 BM25 做 hybrid search（RAGFlow 内置支持） |
| OpenCompass 跑全套耗时 | 抽样模式即可，每个 benchmark 跑 200 题 |
| vLLM 部署 GLM-4-9B-Chat-1M（1M 上下文）需要大量显存 | 上线时按实际场景把 max_model_len 设为 32K-64K |
| LLM-as-Judge 评估有偏 | 关键 checkpoint 必须配人工评估，不能纯靠 GPT 评 |

---

## 七、与"已有进度"的对接建议

如果团队目前在用 ChatGLM3 老仓库或自研脚本：

1. **不要推翻**，先评估现有产出（loss 曲线、checkpoint）
2. **平行起步**：用 LLaMA-Factory 跑一版作为对照基线（1-2 天）
3. **对比 benchmark**（OpenCompass 抽样跑 200 题）
4. **数据说话**：哪边好用哪边，避免技术信仰之争

**智谱官方只提供 1/6 阶段（SFT demo），但开源生态在 2026 已极成熟：用 LLaMA-Factory + Distilabel + RAGFlow + OpenCompass + vLLM 五件套即可覆盖全流水线，全免费且生产级，1 个算法工程师 1-2 周可全栈上手。**
