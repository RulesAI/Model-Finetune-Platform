## 二、核心修正 C1 — Phase 2+3 训练：新增 ms-swift

### 2.2 ms-swift 是什么

| 维度 | ms-swift v4.0 |
|------|---------------|

| 模型仓库 | **ModelScope**（大陆 CDN，下载快 5-10×） |

| 社区活跃度（GitHub stars） | ✅ 更高（国际） | ⚠️ 国内为主 | LLaMA-Factory |

## 三、核心修正 C2 — Phase 4 RAG：新增 LightRAG / GraphRAG

### 3.1 v1 的盲点

v1 直接套了"通用 RAG 推荐"模板（RAGFlow / LangChain / LlamaIndex），**没结合 HR 政策文档的具体特征**——政策条款之间存在大量**强引用网络**，纯向量 RAG 会丢失这种关系。

### 3.2 HR 政策文档的引用网络特征

```
国资委 39 号令《工资总额管理办法》
  ├─ 引用《公司法》第 X 条
  ├─ 配套《国有企业工效挂钩办法》
  ├─ 适用主体：央企（不含金融国企，见单独规定）
  ├─ 例外条款 → 见《科技型企业实施细则》
  └─ 修订历史 → 替代 2008 年原办法

业务问题：央企控股上市公司能否搞股权激励？
答案路径：主体类型(央企控股) → 适用法规(39 号令 §X) → 例外条款(《股权激励管理办法》)
```

纯向量 RAG 在这种**多跳引用**场景下召回率会显著下降——这正是 GraphRAG / LightRAG 解决的问题。

### 3.3 RAGFlow + LightRAG 协同方案

| 层 | 工具 | 职责 |
|----|------|------|
| **文档解析层** | RAGFlow DeepDoc | PDF / docx → 结构化 markdown，表格保留 |
| **向量检索层** | RAGFlow + BGE-M3 + ChromaDB | 语义相似召回（基础底盘） |
| **图增强层** | **LightRAG**（新增） | 提取实体 / 关系 / 引用网络，多跳推理 |
| **Rerank 层** | BGE-reranker-v2-m3 | 融合向量 + 图召回结果，统一打分 |
| **生成层** | 微调后的 GLM-4-9B + system_prompt 注入 | 引用追溯 + 答案合成 |

### 3.4 为什么是 LightRAG 而不是微软 GraphRAG

| 维度 | 微软 GraphRAG | **LightRAG（推荐）** |
|------|--------------|--------------------|
| 速度 | 慢（社区检测 + 全图聚类） | ✅ **快 5-10×**（双层检索，跳过全图聚类） |
| 资源消耗 | 高 | ✅ 中等 |
| 中文支持 | ⚠️ 通用 | ✅ 港大出品，中文实证多 |
| 增量更新 | ⚠️ 需重建 | ✅ 支持增量 |
| 与现有团队栈协同 | ❌ 团队无经验 | ✅ **GraphForge 已用 LightRAG** |

**最关键的协同信号**：用户的 **GraphForge 项目已经全栈使用 LightRAG**（详见 `_供应链-业务工具/GraphForge/CLAUDE.md`「LightRAG 定制原则」章节）。团队已有：

- LightRAG 定制经验（system_prompt / addon_params / llm_model_func 扩展点）
- 与 PostgreSQL 元数据 + Neo4j 图存储的集成模式
- 文档处理 → 抽取 → 入库 → 问答的完整流水线
- LightRAG 0.x 系列踩坑经验

**重复造轮子是反模式**——本项目应直接复用 GraphForge 的 LightRAG 集成层。

### 3.5 落地影响

- `configs/ragflow/` 保留（处理文档解析 + 向量底层）
- 新增 `configs/lightrag/`：图谱构建 + 引用网络抽取配置
- W2 数据工程阶段除了构造 SFT 数据，同时构造 LightRAG 图谱（**并行**，共用文档解析结果）
- 与 GraphForge 团队建立"LightRAG 经验输出"协作机制（1-2 次内部交流即可）

## 四、核心修正 C3 — Phase 6 部署：涉密场景优选 LMDeploy

### 4.1 v1 的盲点

v1 选 vLLM 的理由是"生态最成熟" —— 这对 **SaaS / 多模型混合服务**场景成立，对**国企单机涉密部署**场景是错的。

### 4.2 三大推理引擎在 GLM-4 上的实测对比（2026 H100 数据）

| 引擎 | 吞吐 (tokens/s) | GLM 支持深度 | INT4/FP8 量化 | 信创合规 | 多机部署 | 通用模型支持 |
|------|---------------|------------|--------------|---------|---------|------------|
| **vLLM** | 12,500 | 适配（事后） | ✅ | ⚠️ 国际 | ✅ | ✅ **最全** |
| **SGLang** | 16,200 | ✅ 官方 | ✅ | ⚠️ 国际 | ✅ | ✅ |
| **LMDeploy** ⭐ | 16,200 | ✅ **原生**（上海 AI Lab + 智谱内部用） | ✅ **最强** | ✅ 国产 | ✅ | ⚠️ 略弱 |

### 4.3 选型决策树（v2 正式建议）

```
部署场景？
├── 国企 / 政府 / 涉密 / 单机
│   └── ⭐ LMDeploy（首选）
│       理由：GLM 原生 + INT4 最强 + 信创友好 + 大陆社区
├── 私有云 / 互联网企业 / 多模型混合
│   └── vLLM（首选）
│       理由：生态最全 + 文档丰富 + LangChain/RAGFlow 集成丝滑
└── 极致吞吐 + 多轮对话密集
    └── SGLang（首选）
        理由：RadixAttention 优化 + xAI/AMD 生产验证
```

**HR 国企薪酬场景的明确推荐**：**LMDeploy 作为 Phase 6 主框架**。

### 4.4 落地影响

- `configs/vllm/` 保留作备选（dev 环境快速起步用）
- 新增 `configs/lmdeploy/`：生产部署配置（INT4 量化 + 服务化）
- 端口约定**不变**（仍然 8016 内部，主后端 8006 代理）
- 部署文档区分两套：dev (vLLM) / prod (LMDeploy)

## 八、为什么 v1 漏了这些（复盘）

| 盲点 | 根本原因 | 反思 |
|------|---------|------|
| **C1 漏 ms-swift** | 搜索 keyword 偏国际三巨头（LLaMA-Factory / Unsloth / Axolotl），没专门搜国产生态 | 搜索时必须显式加入"中文 / 国产 / 大陆"维度 |
| **C2 漏 LightRAG** | 套了"通用 RAG 推荐"模板，没结合 HR 政策文档的强引用网络特征 | 工具选型必须**场景化匹配**，不能套通用模板 |
| **C3 高估 vLLM** | 选 vLLM 的理由"生态成熟"对 SaaS 成立，对涉密单机部署是错的 | 客户类型决定部署方案，不是工具决定 |

**这三个盲点的共同特征**：**用通用最佳实践代替了场景特定最佳实践**。

修正后的方法论：

```
工具选型决策的三层匹配：
1. 通用层：工具是否在行业 SOTA 候选集中？（v1 已覆盖）
2. 场景层：客户类型 / 数据特征 / 部署约束是否被工具特化？（v1 漏）
3. 协同层：团队已有工具 / 项目 / 经验能否复用？（v1 漏）
```

**HR 国企 GLM 微调最佳实践 = ms-swift（训练）+ Distilabel（数据）+ LightRAG + RAGFlow（图+向量双层 RAG）+ OpenCompass + QualBench（评估）+ LMDeploy（部署），相比 v1 校正了 3 处场景特化的盲点：国产训练框架 / 图增强 RAG / 涉密单机推理。**
