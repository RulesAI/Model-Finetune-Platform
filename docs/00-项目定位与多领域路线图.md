## 一、项目严格定位

### 1.1 一句话定位

**Model-Finetune-Platform 是一个纯大模型微调工具集**，让算法工程师把"行业素材 → 微调好的模型 + 推理 API"端到端跑通。通过**场景插件机制**支持多个行业领域（HR / 制造 / 供应链 / ...）。

### 1.2 我们解决什么问题

| 痛点 | 当前业界解法 | 本项目解法 |
|------|------------|----------|
| 微调工具栈分散（训练 / 评估 / 部署各一套）| 算法工程师每次自己拼装 | 整合 ms-swift + Distilabel + OpenCompass + LMDeploy 主流栈 |
| 行业适配重复造轮子（HR 一套、制造一套）| 每个领域单独建项目 | **场景插件**机制，80% 通用 + 20% 场景特定 |
| 智谱官方仅 SFT demo（1/5 阶段）| 各自摸索 | 覆盖全微调流水线 + 涉密本地化 |
| 国企客户要信创合规 | 国际工具栈风险 | 国产首选（ms-swift / LMDeploy）|

### 1.3 不做什么（严守边界）

| ❌ 不做 | 为什么 | 归属 |
|--------|--------|------|
| **RAG / 知识图谱** | 不属于微调技术内涵 | **GraphForge 项目主业** |
| 业务系统集成 / Agent 编排 | 工具集不是业务系统 | 下游业务系统（PayLens / Nexus / SCI.AI / 自建 Agent）|
| 数据标注平台 | 集成现成的就够 | 集成 Argilla |
| 基础模型预训练 | 不在能力 / 算力范围 | 模型供应商（智谱 / 阿里）|
| 多模态训练 | v1 优先文本场景 | v2+ 再考虑 |
| 模型 hosting SaaS 服务 | v1 私有化部署 | 未来商业化时考虑 |

### 1.4 用户角色明确

```
内部用户（本工具集的直接服务对象）
├── 算法工程师 → 跑 5 阶段流水线，调超参，出模型
└── 实施工程师 → 用工具集为新客户/新场景快速复制能力

外部消费者（不直接接触工具集）
└── 下游业务系统 → 调用工具集产出的模型 API
    ├── HR PayLens / HR Knowledge Nexus（HR 系列）
    ├── GraphForge（RAG 服务的 LLM 后端）
    ├── SCI.AI / Supply Nexus Agent（供应链系列）
    └── 自建业务 Agent

业务方（终端用户，不直接接触工具集）
├── HR 部门
├── 制造车间
└── 供应链运营
（业务方使用上面的业务系统，业务系统调用本项目模型 API）
```

## 二、两层架构：通用工具集 + 场景插件

### 2.1 设计原则

**80/20 法则**：微调流水线中 80% 领域无关（数据清洗、SFT 算法、评估 benchmark、推理服务化），20% 领域特定（指令模板、领域评估、提示词）。

把 80% 沉淀到 `core/` + `configs/` + `scripts/`，把 20% 隔离到 `scenes/<scene>/`，得到：

```
        ┌─────────────────────────────────────┐
        │   通用工具集 (Core, 领域无关)        │
        │   ├── Phase 1-5 工具封装             │
        │   ├── pipeline 脚本（接受 --scene）  │
        │   └── 评估 / 部署基础设施            │
        └──────────────────┬──────────────────┘
                           │
                ┌──────────┼──────────┐
                ▼          ▼          ▼
        ┌──────────┐ ┌──────────┐ ┌──────────┐
        │ HR-salary│ │  Mfg     │ │ Supply-  │
        │ 场景插件 │ │  场景插件│ │ chain    │
        └──────────┘ └──────────┘ └──────────┘
        • prompts/        • prompts/      • prompts/
        • instructions/   • instructions/ • instructions/
        • eval_set/       • eval_set/     • eval_set/
        • system_prompts/ • ...           • ...
```

### 2.2 通用工具集（5 阶段）

| 阶段 | 工具 | 跨领域通用性 |
|------|------|------------|
| Phase 1 数据 | MarkItDown / Distilabel / Presidio | 100% 通用 |
| Phase 2 SFT | ms-swift / LLaMA-Factory / Unsloth | 100% 通用 |
| Phase 3 DPO | ms-swift | 100% 通用 |
| Phase 4 评估 | OpenCompass / DeepEval | 通用（领域 benchmark 可插拔）|
| Phase 5 部署 | LMDeploy / vLLM / LiteLLM | 100% 通用 |

### 2.3 场景插件（领域特定）

每个场景目录包含 4 类内容：

```
scenes/<scene>/
├── README.md             # 场景定位 / 数据来源 / 触发时机
├── prompts/              # Distilabel 数据增广用的领域 prompt
├── instructions/         # SFT 指令构造规则
├── eval_set/             # 场景评估集
└── system_prompts/       # 推理时的领域 system prompt
```

### 2.4 复用收益

新增一个领域（如生产制造）的工作量：

| 维度 | 不用插件架构 | 用插件架构（本项目） |
|------|------------|------------------|
| 复制核心代码 | ✅ 完整复制 | ❌ 不需要 |
| 修改训练框架 | ✅ 改 | ❌ 配置参数即可 |
| 写领域数据规则 | ✅ 必做 | ✅ 必做（在 scenes/）|
| 写领域评估集 | ✅ 必做 | ✅ 必做（在 scenes/）|
| 工程师 onboarding | 1-2 周 | **2-3 天** |
| 总工作量 | 4-6 周 | **1-2 周** |

## 三、最终产出（"三件套"，v3 收窄）

每个场景跑完 5 阶段后，本工具集向**下游业务系统**交付：

```
┌─────────────────────────────────────────────────┐
│   <场景> 微调模型交付包（不含 RAG）              │
├─────────────────────────────────────────────────┤
│  ① 微调模型                                      │
│     - GLM-4-9B-Chat-1M + 场景 LoRA adapter       │
│     - LMDeploy INT4 量化版                       │
│                                                  │
│  ② 推理 API（OpenAI 兼容）                       │
│     - LMDeploy 服务化部署                        │
│     - 任意业务系统可直接调用                     │
│                                                  │
│  ③ 配套资产                                      │
│     - 评估报告（C-Eval / CMMLU / 领域 benchmark）│
│     - System Prompt 模板（推理使用）             │
│     - 运维手册 + SLA                             │
└─────────────────────────────────────────────────┘
```

### 3.1 不在交付物中的（v2 → v3 移除）

| 移除项 | 现在归属 |
|--------|---------|
| RAG 服务（向量 / 图增强）| **GraphForge 项目交付** |
| 知识图谱构建工具 | GraphForge 主业 |
| Agent 编排 | 下游业务系统自建 |

### 3.2 下游业务系统怎么用本项目产出

```
下游业务系统（如 HR PayLens 或 GraphForge）
  ↓ 1. 拿到本项目交付的 API 端点
  ↓ 2. 拿到 System Prompt 模板
  ↓ 3. 拿到评估报告作背书
  ↓
  └── 业务系统自己决定：
       • 直接用模型回答（不需要 RAG 的场景）
       • 接 GraphForge RAG 做检索增强（需要事实精度的场景）
       • 加 Agent 编排做多步骤业务（复杂场景）
```

**本工具集不关心下游怎么用，只保证模型 + API 质量。**

## 六、与既有项目的协同关系

### 6.1 上下游关系图

```
┌──────────────────────────────────────────────────────────────┐
│              Model-Finetune-Platform（本项目）                │
│              产出：微调模型 + OpenAI 兼容 API                 │
└─────────────────────────┬────────────────────────────────────┘
                          │ 模型 API
                          ▼
   ┌──────────────────────────────────────────────────────┐
   │                  下游消费方                            │
   ├──────────────────────────────────────────────────────┤
   │  GraphForge          → RAG 服务的 LLM 后端           │
   │  HR PayLens          → HR 报告撰写的 LLM 后端        │
   │  HR Knowledge Nexus  → HR 知识问答的 LLM 后端        │
   │  Supply Nexus Agent  → 供应链问答的 LLM 后端         │
   │  EvoChain-Agent      → 内容生成的 LLM 后端           │
   │  自建业务 Agent      → 各类业务的 LLM 后端           │
   └──────────────────────────────────────────────────────┘
```

### 6.2 与 GraphForge 的明确边界

| 职责 | 归属 |
|------|------|
| 知识图谱构建（实体抽取 / 关系建模）| **GraphForge** |
| 向量库 + 图增强 RAG（LightRAG）| **GraphForge** |
| 模型微调（让 LLM 学领域语言）| **本项目** |
| 微调后的模型部署 + API | **本项目** |
| 业务系统集成 | 业务项目（PayLens / Nexus 等）|

**简言之**：
- 本项目让 LLM **掌握**领域语言（改权重）
- GraphForge 让 LLM **看见**领域知识（不改权重，外挂知识库）
- 两者通过"模型 API"接口对接，**职责零重叠**

### 6.3 GraphForge 可能调用本项目模型 API

GraphForge 现在用 SiliconFlow API（DeepSeek-V3.2）作为 LLM 后端。未来可：
1. 本项目跑完 HR-salary 微调
2. 部署 API 在 8016
3. GraphForge 切换 LLM 后端到本项目 API
4. GraphForge 仍负责 RAG / 知识图谱
5. **双方协同产出端到端 HR AI 助手给客户**

## 七、关键架构决策（ADR）

### ADR-001：GLM-4-9B-Chat-1M 作为首选基座

理由：HR 客户偏好国产 + 1M 上下文 + 智谱品牌

### ADR-002：通用工具集 + 场景插件两层架构

理由：客户领域将扩展，避免每领域复制代码

### ADR-003：ms-swift 作为国企场景训练框架首选

理由：GLM 一等支持 + ModelScope CDN + 信创友好

### ADR-004：~~RAG 双层架构（向量 + 图增强）~~ → **方案 A 严格定位收窄**

**v3 撤销**：v2 推荐"RAGFlow + LightRAG"双层 RAG。v3 严格定位"纯微调工具集"后，RAG 整体划给 GraphForge，本项目不实现。

### ADR-005：涉密单机推理用 LMDeploy

理由：GLM 原生 + INT4 量化最强 + 信创友好

### ADR-006（v3 新增）：用户角色严格收窄为算法/实施工程师

理由：之前混淆"业务方/客户/工程师"，导致 RAG 被纳入工具集。收窄后职责清晰，5 周交付节奏可保。

### ADR-007（v3 新增）：交付物从"四件套"调整为"三件套"

理由：移除 RAG 服务（划给 GraphForge），保留模型 + API + 配套资产。

## 十、一句话摘要
