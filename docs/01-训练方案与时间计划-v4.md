## 二、CPT 触发条件（v4 新增）

### 2.1 是否做 CPT 决策树

```
场景数据量？
├── < 1 亿 token       → 跳过 CPT（v3 默认路径）
│                        ↓
│                        Phase 1 → 2 → 3 → 4 → 5
│
└── ≥ 1 亿 token       → 必做 CPT（v4 新增路径）
                          ↓
                          Phase 0 (CPT) → 1 → 2 → 3 → 4 → 5
```

### 2.2 当前两场景路径

| 场景 | 数据量 | 是否 CPT | 流水线阶段 |
|------|--------|---------|----------|
| **HR-salary** | 140 万 token | ❌ 不做 | 5 阶段（v3 路径，不变）|
| **TCM** | ~8-10 亿 token | ✅ **必做** | **6 阶段**（v4 新增 Phase 0）|
| Manufacturing | TBD | 待数据画像 | 待决定 |
| Supply-chain | TBD | 待数据画像 | 待决定 |

### 2.3 为什么 CPT 阈值定 1 亿 token

| 阈值 | 含义 |
|------|------|
| < 1 亿 token | CPT 几乎学不到东西，是"数据噪声 + 灾难性遗忘"双重风险 |
| 1 亿 - 10 亿 token | CPT 可注入领域核心知识，但学到的概念不深 |
| ≥ 10 亿 token | CPT 完整训练领域知识体系，效果显著 |

**保守策略**：1 亿 token 触发，到 10 亿 token 效果最佳。

## 三、Phase 0 CPT 流水线（v4 新增）

### 3.1 工具选型

| 子任务 | 工具 | 配置 |
|--------|------|------|
| 数据预处理 | 自写脚本 + Datasets 库 | streaming mode，避免内存爆炸 |
| Tokenization | ms-swift 内置 | GLM-4-9B 自带 tokenizer |
| CPT 训练 | **ms-swift CPT 命令** | `swift pt` 子命令（已支持）|
| 多机多卡 | **DeepSpeed ZeRO-3**（不用 Unsloth，F1 校正）| 8 卡 H100 起步 |
| 训练监控 | SwanLab + tensorboard | loss + ppl 双监控 |
| Checkpoint 保存 | 每 1000 步 / 每小时 | 防中断 |

### 3.2 CPT 训练命令（ms-swift v4）

```bash
swift pt \
  --model_id_or_path ZhipuAI/glm-4-9b-chat-1m \
  --train_type full \
  --dataset data/cpt/tcm/corpus.jsonl \
  --output_dir checkpoints/tcm/cpt-v1 \
  --learning_rate 2e-6 \
  --num_train_epochs 1 \
  --deepspeed zero3 \
  --max_length 32768
```

**关键参数**：
- 学习率 **2e-6**（CPT 比 SFT 低一个量级，防灾难性遗忘）
- Epoch **1**（CPT 通常 1-3 epoch，TCM 数据多，1 epoch 起步）
- DeepSpeed ZeRO-3（多机分片）
- max_length 32K（古籍长文本支持）

### 3.3 CPT 关键约束

| 约束 | 原因 |
|------|------|
| 起步于 instruct 模型（非 base）| ChatGLM3 / StatLLaMA 研究：base 起步后续 SFT 救不回来 |
| 极低学习率 2e-6 | 防灾难性遗忘 |
| 混入 5-10% 通用语料 | 同样防遗忘（通用回放）|
| C-Eval / CMMLU 持续监控 | 每 epoch 末跑 200 题，掉点 > 5% 立即停 |
| 长文本截断 32K | 古籍单卷可能 5-10 万字，必须 32K+ |

### 3.4 CPT 阶段评估

| 指标 | 工具 | 通过线 |
|------|------|--------|
| 训练 loss | tensorboard | 单调下降 |
| 验证集 PPL | ms-swift 内置 | 下降至 base 的 60-70% |
| C-Eval / CMMLU 通用能力 | OpenCompass | 掉点 < 5%（CPT 比 SFT 容忍度高一点）|
| TCM benchmark（自建 / 中医执业医师题）| OpenCompass | 比 base 提升 ≥ 10% |

## 四、F1-F4 校正落实

### 4.1 F1 — Unsloth 使用约束

**在 `configs/ms_swift/scenes/<scene>/sft.yaml` 默认配置**：

```yaml
# Unsloth backend 启用条件（v4 F1 校正）
# 仅"单卡 + LoRA + 数据量 < 1 亿 token"场景启用，否则注释掉
# 适用：HR-salary (140 万 token, 单卡)
# 不适用：TCM (8 亿+ token, 多卡 DeepSpeed)
use_unsloth: true   # HR 场景
# use_unsloth: false  # TCM 场景请注释 + 启用 deepspeed: zero3
```

**在 `scripts/training/sft.py` 加运行时检查**：

```python
def check_unsloth_eligibility(scene_config):
    if scene_config['use_unsloth']:
        if scene_config['estimated_tokens'] > 100_000_000:
            raise ValueError(
                f"Scene {scene_config['scene']} has {scene_config['estimated_tokens']} tokens, "
                "exceeds Unsloth 1B threshold. Disable use_unsloth in config."
            )
```

### 4.2 F2 — QualBench 前置 baseline

**Phase 4 评估流程更新**：

```
Step 4.0 (v4 新增)：跑 base 模型 baseline
  ├── 对象：GLM-4-9B-Chat-1M 原始权重（未微调）
  ├── 数据集：QualBench HR 资格类 100 题（HR 场景）/ 中医执业医师 500 题（TCM 场景）
  └── 输出：logs/eval/<scene>/baseline_<benchmark>.md

Step 4.1：评估微调后模型
  ├── 对比 baseline
  ├── 提升 < 10% → 微调效果不显著，需重训
  ├── 提升 ≥ 15% → 合格
  └── 提升 ≥ 20% → 优秀
```

### 4.4 F4 — Presidio 中文 PII 字典前置

**目录结构调整**：

```
scenes/<scene>/
└── pii_rules/
    ├── chinese_common.yaml    # 通用中文 PII（HR/TCM/制造/供应链 共用）
```
