# 03-场景扩展指南（v3 收窄版）

> 创建：2026-05-25
> v3 变化：删除 RAG 相关步骤（划给 GraphForge），6 步流程聚焦"纯微调"

---

## 一、何时启动新场景

满足以下**全部**条件才启动：

- [ ] 业务方明确客户与商业价值
- [ ] 客户已确认提供素材（或 1 个月内可提供）
- [ ] 素材规模满足训练阈值（至少 3000 条可衍生 instruction 数据）
- [ ] 团队有 1 名算法工程师 + 1 名领域专家可投入 ≥ 2 周

未满足时：场景目录可建占位（`scenes/<scene>/README.md` 仅写路线图），但**不投入开发**。

## 二、新场景启动 6 步流程

### Step 1：填写场景数据画像（必做）

```yaml
# scenes/<scene>/data_profile.yaml
scene_name: <scene>
client_type: <国企 / 民企 / 政府 / ...>
data_sources:
  - name: <数据源名称>
    path: <软链路径>
    size: <bytes>
    file_count: <int>
    estimated_tokens: <int>
topic_structure: <子目录 / 分类>
sensitivity: <high / medium / low>
multimodal_required: <true / false>   # 决定是否需切多模态基座
# v3 移除字段：reference_network_density（RAG 划给 GraphForge）
downstream_consumers:                  # v3 新增：列出下游业务系统
  - <消费方 1>（如 HR PayLens）
  - <消费方 2>（如 GraphForge RAG 后端）
```

### Step 2：创建场景目录骨架

```bash
# 复制 HR-salary 作为模板
cp -r scenes/hr-salary scenes/<new-scene>
# 清空内部内容
find scenes/<new-scene> -type f ! -name 'README.md' ! -name '.gitkeep' -delete

# 复制工具配置模板（v3 不再含 ragflow / lightrag）
for tool in ms_swift llama_factory distilabel opencompass lmdeploy; do
  cp -r configs/$tool/scenes/hr-salary configs/$tool/scenes/<new-scene>
done
```

### Step 3：填充 `scenes/<new-scene>/` 4 类内容

#### 3.1 `prompts/`（指令构造模板）

```
prompts/
├── qa.jinja          # 问答模板
├── generation.jinja  # 方案生成模板
└── extraction.jinja  # 信息抽取模板
```

#### 3.2 `instructions/`（数据合成配方）

```yaml
# instructions/rules.yaml
seed_examples_count: 50
distillation_count: 15000
distribution:
  qa: 0.50
  generation: 0.30
  extraction: 0.20
teacher_model: claude-4.6
seed_path: instructions/seed_examples.jsonl
output_path: ../../data/instructions/<scene>/sft_v1.jsonl
```

#### 3.3 `eval_set/`（评估集）

```
eval_set/
├── benchmark.jsonl   # 200-500 条业务测试集
└── golden.jsonl      # 黄金答案
```

#### 3.4 `system_prompts/`（推理 system prompt）

```markdown
# system_prompts/default.md
你是 {{客户名称}} 的 {{领域}} 智能助手。
回答时遵守：
1. 严格按行业惯例与语言风格回答
2. 涉及具体数字 / 法规 / 条款时，明确说"需要查阅原文确认"
3. 不确定的内容明确说"需要人工确认"
```

> **v3 关键**：system_prompt 中"事实精度"由下游业务系统通过 RAG 实现，本项目不承担。

### Step 4：定制工具配置

#### 4.1 `configs/ms_swift/scenes/<new-scene>/sft.yaml`

```yaml
model_id_or_path: ZhipuAI/glm-4-9b-chat-1m
dataset: data/instructions/<new-scene>/sft_v1.jsonl
output_dir: checkpoints/<new-scene>/sft-v1
```

#### 4.2 `configs/opencompass/scenes/<new-scene>/eval.yaml`

```yaml
benchmarks:
  - ceval        # 通用（必跑）
  - cmmlu        # 通用（必跑）
  - <domain_benchmark>  # 领域特定
business_test_set: scenes/<new-scene>/eval_set/benchmark.jsonl
```

#### 4.3 `configs/lmdeploy/scenes/<new-scene>/serve.yaml`

```yaml
model_path: checkpoints/<new-scene>/dpo-v1
quant_policy: 4
server_port: 8016
```

> **v3 移除**：~~configs/ragflow/scenes/...~~ + ~~configs/lightrag/scenes/...~~ 不再需要。

### Step 5：跑通 5 阶段流水线

```bash
# v3 仅 5 阶段
python scripts/data_prep/build_sft_dataset.py --scene <new-scene>
swift sft --config configs/ms_swift/scenes/<new-scene>/sft.yaml
swift rlhf --rlhf_type dpo --config configs/ms_swift/scenes/<new-scene>/dpo.yaml
python scripts/evaluation/run_eval.py --scene <new-scene>
lmdeploy serve api_server --config configs/lmdeploy/scenes/<new-scene>/serve.yaml --port 8016
```

> **v3 移除**：~~python scripts/data_prep/build_lightrag_kg.py~~ 不再执行（RAG 归 GraphForge）

### Step 6：评估与对下游消费方交付

- 复用 `scripts/evaluation/run_eval.py`
- 输出报告到 `logs/eval/<new-scene>/`
- 把 API 端点、System Prompt 模板、评估报告交付给下游业务系统（消费方）
- **下游消费方**（GraphForge / PayLens / Nexus / Agent）自行决定是否接 RAG

## 三、新场景预期时间表（v3 提速）

| 阶段 | 时间 | 责任人 |
|------|------|--------|
| Step 1 数据画像 | 0.5 天 | 业务 + 算法 |
| Step 2 目录骨架 | 0.5 天 | 算法 |
| Step 3 场景内容 | 3-5 天 | 算法 + 领域专家 |
| Step 4 工具配置 | 0.5 天 | 算法 |
| Step 5 跑通流水线 | 4-6 天 | 算法 |
| Step 6 评估与下游交付 | 2-3 天 | 算法 |
| **合计** | **10-15 天**（约 2-3 周）| - |

vs v2 的 14-21 天（约 3 周），**v3 收窄后再缩短约 4-6 天**。

## 四、领域差异点速查表（v3）

| 维度 | HR-Salary | Manufacturing | Supply-Chain |
|------|-----------|--------------|--------------|
| 文档类型 | 政策 / 案例 / 书籍 | 操作规程 / SOP / 故障树 | 业务流程 / 单据 |
| 数据形态 | 纯文本 | 可能含图纸（v2+ 多模态）| 半结构化 |
| 多模态需求 | 否 | **可能需** | 否 |
| 实体类型 | 政策 / 条款 / 主体 | 设备 / 工艺 / 故障 | 流程 / 单据 |
| 评估特色 | QualBench HR 资格 | 自建制造 benchmark | 业务流程合规 benchmark |
| 主要下游消费方 | HR PayLens / Nexus / GraphForge | （待定）| Supply Nexus Agent / GraphForge |

> **v3 移除**：v2 版本里"引用网络密度"、"是否需 LightRAG" 列已移除（与本项目无关，归 GraphForge 评估）。

## 五、避免的常见反模式

| 反模式 | 正确做法 |
|--------|---------|
| ❌ 在 `scripts/` 写场景特定逻辑 | ✅ 通过 `--scene` 参数从 `scenes/<scene>/` 读 |
| ❌ 把 prompt 硬编码到 Python | ✅ 放 `scenes/<scene>/prompts/*.jinja` |
| ❌ 一份 system prompt 复用所有场景 | ✅ `scenes/<scene>/system_prompts/default.md` |
| ❌ 不写场景路线图就开始填代码 | ✅ Step 1 数据画像必做 |
| ❌ 没有领域 benchmark 就上线 | ✅ Step 3 必须出 `eval_set/benchmark.jsonl` |
| **❌ 在 scenes/ 实现 RAG 逻辑** | ✅ **转告 GraphForge 团队**（v3 严格定位）|

## 六、Scene Checklist（启动前 / 上线前）

### 启动前（Day 0）

- [ ] 业务方确认数据来源 / 规模 / 时间表
- [ ] 数据画像 yaml 已填
- [ ] 涉密评估通过
- [ ] 资源到位（算法 + 领域专家 + GPU）
- [ ] **下游消费方明确**（哪些业务系统会调用本场景模型）

### 上线前（Day 10-15）

- [ ] 5 阶段流水线跑通
- [ ] 通用能力 C-Eval / CMMLU 掉点 < 3%
- [ ] 领域 benchmark 通过率 ≥ 业务方约定阈值
- [ ] 业务测试集人工评估 100 条 ≥ 4.0/5
- [ ] LMDeploy 部署稳定（P95 < 3s / 24h 不掉）
- [ ] **下游消费方**已知晓 API 端点 + System Prompt 模板
- [ ] 业务方签字验收（本项目交付物 = 模型 + API + 评估报告 + System Prompt）
