## 三、为什么 v2 → v3 收窄

### 3.1 定位混淆复盘

v2 的问题：把"工具集"和"端到端解决方案"混为一谈。

| 层次 | v2 误以为 | v3 正确认知 |
|------|---------|-----------|
| 工具集本身 | 包含 RAG | 仅微调 |
| 端到端方案 | = 工具集 | = 工具集 + GraphForge + 业务系统 |
| 客户视角 | 工具集要直接对客户 | 工具集对内部工程师，端到端方案对客户 |

### 3.2 严格定位的收益

| 维度 | v2 | v3 |
|------|----|----|
| 工具栈复杂度 | 20+ 工具 | **12 工具**（减 8）|
| 5 阶段 vs 6 阶段 | 6 阶段 | **5 阶段** |
| 交付时间 | 6 周（首发）/ 3-4 周（后续）| **5 周**（首发）/ **2-3 周**（后续）|
| 与 GraphForge 职责重叠 | 部分重叠（都做 RAG）| **零重叠** |
| 团队 onboarding | 工具多易学习成本高 | **更聚焦** |

## 五、删除工具的归属说明（与 GraphForge 接口）

### 5.1 RAGFlow / LightRAG / BGE-M3 全部归 GraphForge

GraphForge 已经全栈使用 LightRAG（见主 CLAUDE.md「LightRAG 定制原则」一节），并：
- 集成 PostgreSQL + Neo4j
- 有 system_prompt 扩展点
- 有完整文档处理流水线

**本项目不重造，与 GraphForge 通过"模型 API"接口对接**。

### 5.2 协作模式（v3 明确）

```
本项目产出：微调模型 + OpenAI 兼容 API（端口 8016）
                            │
                            ▼
GraphForge 配置：把 LightRAG 的 llm_model_func 改为调用本项目 API
                            │
                            ▼
GraphForge 产出：RAG 服务（含本项目模型作 LLM 后端）
                            │
                            ▼
业务系统集成：调用 GraphForge RAG 接口
                            │
                            ▼
最终客户产品：行业 AI 助手
```

### 5.3 端到端项目交付的协同分工

当业务方需要"端到端 HR AI 助手"时：

| 项目 | 交付物 | 时间 |
|------|--------|------|
| **本项目** | 模型 + API + 评估报告 + System Prompt 模板 | 5 周 |
| **GraphForge** | RAG 服务（消费本项目 API） | 与本项目并行 / 或之后 1-2 周 |
| **下游业务系统**（PayLens 等）| 业务集成 + UI | 2-3 周 |
| **总交付周期** | 6-8 周（端到端） | - |

注意：**总交付周期不是本项目的承诺**。本项目只承诺自己那 5 周。

## 八、风险与避坑清单（v3 简化）

| 风险 | 规避策略 |
|------|---------|
| 业务方要求工具集做 RAG | 明确边界，转告 GraphForge |
| ms-swift 配置文件踩坑 | 直接复制 `examples/train/glm4_sft.yaml` |
| ms-swift 与 GLM-4 不完全兼容 | LLaMA-Factory 兜底 |
| 通用能力遗忘 | C-Eval / CMMLU 抽 200 题，每轮训练后跑 |
| LMDeploy 部署 GLM 1M 上下文 OOM | 部署时设 max_model_len 32K-64K |
| LLM-as-Judge 评估有偏 | 关键 checkpoint 必须人工评估 |
