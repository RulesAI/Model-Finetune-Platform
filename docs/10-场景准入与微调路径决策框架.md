# 场景准入与微调路径决策框架

> **本文定位**：给 Model-Finetune-Platform 的「场景准入」与「微调路径选择」一套可复用的判据。
> 解决的问题：当一个新垂直场景找上门，如何客观判断 **它到底该不该进本（LLM 微调）平台**，
> 还是该被分流给 CV 训练线 / GraphForge（RAG）/ 规则引擎；若确实要微调，又该走哪条路径。
>
> **来源**：2026-06-13 围绕某航空制造客户的「焊接件射线检测（RT）智能质检」AI 赋能方案的一次深度探讨，
> 把"垂直领域要不要微调"这个反复出现的伪命题彻底拆开。完整探讨过程见本文 §6 附录（探讨实录）。
> （客户具名与售前案例细节见内网项目库，本文脱名为通用范例；下文以「航空 RT 质检范例」指代该案例。）
>
> **一句话核心论点**：**"垂直领域 ⇒ 要微调" 是错的判据。决定要不要微调的，从来不是行业名字或领域深浅，
> 而是「这块领域智能能不能被规则/检索/CV/解析器接住」+「LLM 的输出可不可验证」+「为什么微调（领域驱动还是部署驱动）」。**

---

## 0. 为什么需要这套判据

本平台严格定位「纯大模型微调工具集」（见 [`CLAUDE.md`](../CLAUDE.md) 与 [`00-项目定位与多领域路线图.md`](00-项目定位与多领域路线图.md)）。
最大的范围风险不是"做不出"，而是**接进一堆"其实根本不需要微调"的伪需求**——
把 CV 感知任务、RAG 检索任务、可编码的规则判定任务，都当成"要微调大模型"硬塞进来，
既稀释交付速度，又违背"5 周交付首发 / 3 周交付后续"的根本约束。

这套判据就是平台的**入口闸门**：一个场景来了，先用它分流，再决定要不要动微调工具链。

---

## 1. 第一性原理：领域智能"装在哪里"

任何垂直场景落地，领域知识必然存在，问题只是它**落在哪个组件**。这是全部判断的地基：

| 领域智能的落点 | 对应组件 | 要不要微调 LLM | 归属 |
|---|---|---|---|
| 事实 / 条款 / 文档内容 | RAG / 向量检索 | 否（检索时注入） | **GraphForge** |
| 确定性规则 / 已成文标准 | 规则引擎 / KG | 否 | 规则引擎 / KG |
| 几何 / 结构化文件（CAD/STEP） | 解析器 + 规则 | 否 | 工程解析 |
| 视觉感知（图像缺陷/分割/检测） | CV 模型训练 | **CV 要训，但不是 LLM 微调** | **CV 训练线** |
| 格式 / 模板 | 模板引擎 | 否 | 模板 |
| **隐性判断 / 领域推理 / 领域风格，且以上都接不住** | **只能进 LLM 权重** | ✅ **这才是本平台的活** | **Model-Finetune-Platform** |

> **判据**：领域智能能被外置（规则/检索/CV/解析/模板）→ 外置，不微调；
> 外置不了、必须长进权重 → 才微调。**看的不是行业，是这块知识的"可外置性"。**

---

## 2. 关键澄清：三个最常踩的认知误区

探讨中反复确认的三条，写进判据前必须先校准：

### 误区一：「垂直领域 ⇒ 需要微调」
**错。** 一个领域可以极深（如航空 NDT），但如果它的核心知识已经成文为标准（可编码）、
要求强可审计（禁止黑盒）、核心任务是感知（CV 的活），那么最佳实践恰恰是**把智能外置、不微调 LLM**。
**越是深垂直 + 强监管，核心反而越倾向于"不靠 LLM 微调"**（见 §3 神经符号范式）。

### 误区二：「用 LLM ⇒ 要微调 LLM」
**错。** 泛化是 LLM 的**原生能力**（预训练给的），通过 Prompt + 架构（解析器/KG/RAG）就能调用出来，**不微调也有**。
微调不是为了"获得泛化"，而是为了"适配领域格式/术语"或"在小模型上恢复能力"——
且重度微调会**伤泛化**（过拟合、灾难性遗忘）。
**想要"对新输入泛化" → 用 LLM；不等于 → 微调 LLM。**

### 误区三：「微调就是微调，没区别」
**错。** 微调要再分**动机**，因为动机决定工具链、数据、评估方式完全不同（见 §4 Step 3）：
- **领域驱动**：LLM 是产品，要会判断/写/推理（HR 薪酬、TCM 辨证）；
- **部署驱动**：涉密/内网逼着用小模型，微调/蒸馏是为了把通用能力"恢复"到可用线（航空 RT 质检范例 M00 文档理解若顶不住）。

## 3. 两种垂直、两种最佳实践

| | **A 类：可编码 + 强监管 + 感知型** | **B 类：隐性知识 + 生成判断型** |
|---|---|---|
| 典型 | 航空/医疗影像检测、工业质检、安全攸关判定 | 报告撰写、咨询、法律意见、中医辨证 |
| 知识形态 | 已成文标准、可编码规则、感知靠 CV | 隐性、专家说不清、要整体判断/生成 |
| 最佳实践 | **神经符号 / 混合 AI**：CV 感知 + 规则/KG 判定，LLM 当薄工具 | **学进权重**：领域微调，LLM 当大脑 |
| 要不要微调 LLM | 基本不要（外置） | 要 |
| 代表 | 航空 RT 质检范例（见 §5） | 本平台首发 HR 薪酬 / TCM |

> **航空 RT 质检范例是 A 类的教科书**：等级判定→规则引擎、缺陷感知→CV(YOLO/SAM)、坐标→解析器、标准→KG+检索、报告→模板，
> 领域智能被系统性外置，留给 LLM 的只剩"读文档抽字段"的搬运活——所以**不需要微调，不是因为领域浅，是因为智能被故意放到了别处**。

## 4. 决策流程（场景准入 + 微调路径）

### 4.1 流程图

```
            ┌─────────────────────────────────────────────┐
   Step 0   │  把场景拆成"能力模块"，逐模块判              │
            │  （感知/判定/生成/检索/抽取——别整场景一刀切）│
            └───────────────────────┬─────────────────────┘
                                     │ 对每个模块：
                                     ▼
   Step 1   ┌─────────────────────────────────────────────┐
  能不能    │  这块领域智能能被"外置"吗？                  │
  外置？    └───────────────────────┬─────────────────────┘
                │可编码规则/标准 → 规则引擎/KG ───────────┐
                │事实/条款检索   → RAG/向量(→GraphForge) ─┤  不进本平台
                │几何/CAD/STEP   → 解析器+规则 ───────────┤  （分流出去）
                │视觉感知        → CV 训练(→CV 线) ───────┘
                │
                │ 都接不住（隐性判断/领域生成/推理）
                ▼
   Step 2   ┌─────────────────────────────────────────────┐
  用≠微调   │  先别急微调：Prompt+few-shot+RAG+结构化输出  │
            │  打到什么程度？输出可验证/可逆吗？          │
            └───────────────────────┬─────────────────────┘
                │输出可验证/可逆(有试拍/审核兜底) → 原生泛化够用，先不微调
                │便宜修法(OCR/版面/术语表)未尽 → 先修上游，不微调
                │
                │ Prompt 到顶仍不达标 & 修法已尽 & ROI 撑得起
                ▼
   Step 3   ┌─────────────────────────────────────────────┐
  为什么    │  确定要微调——分清动机（决定工具链/评估）    │
  微调？    └───────────────────────┬─────────────────────┘
                │领域驱动(LLM是产品)  → 领域 eval_set + 风格 win-rate + SFT/DPO
                │部署驱动(小模型恢复) → 蒸馏 + 能力保持率评估 vs 教师模型
                ▼
            ┌─────────────────────────────────────────────┐
   Step 4   │  微调前最后一道 ROI 闸：                     │
            │  微调+数据成本 < 省下的人力/质量收益？       │
            │  否 → 退回 Prompt 方案；是 → 进平台微调流程  │
            └─────────────────────────────────────────────┘
```

### 4.2 决策表（速查）

| Step | 问题 | 判据 / 出口 |
|------|------|------------|
| **0 拆模块** | 场景是一个模型任务吗？ | **不是。** 先按能力（感知/判定/生成/检索/抽取）拆，逐模块走下面的流程 |
| **1 可外置性** | 这块智能能被规则/检索/CV/解析接住吗？ | 能 → **分流出平台**（规则引擎 / GraphForge / CV 线 / 解析器）；不能 → 进 Step 2 |
| **2 用≠微调** | Prompt/few-shot/RAG 打到顶了吗？输出可验证可逆吗？ | 可验证可逆（有兜底）或便宜修法未尽 → **先不微调**；Prompt 到顶仍不达标 → 进 Step 3 |
| **3 微调动机** | 为什么要微调？ | **领域驱动** → SFT/DPO + 领域 eval；**部署驱动** → 蒸馏 + 能力保持率 |
| **4 ROI 闸** | 微调成本 < 收益？ | 否 → 退回 Prompt；是 → **正式进平台微调流程** |

### 4.3 关键阈值与红线

- **可验证可逆的输出，放手让 LLM 原生泛化做**（输出是"假设/草案"，下游有试拍/仿真/人工审核兜底，错了便宜可逆）。
- **最终、不可逆、要担责的输出，绝不交黑盒 LLM**（要么规则化、要么人工签署），尤其安全攸关/适航场景。
- **微调前必过"便宜修法闸"**：OCR/版面 → 文档 AI；缺矢量源 → 数字化；可编码 → 规则；偶尔错且有兜底 → 不微调。
- **正确性不该靠微调来保**：有验证兜底时，微调只买"更少的人工返工"，不买"正确性"——所以微调决策本质是 **ROI 问题**，不是"模型能不能做"问题。

## 5. A 类范例全模块走查（航空 RT 质检）

把决策框架套到该范例场景的 11 个模块（M00–M10），逐格分流：

| 模块 | 任务 | Step 1 落点 | 进本平台? | 说明 |
|------|------|------------|:---:|------|
| M03 缺陷检测（核心） | YOLO v8 检测 6 类缺陷 | CV 感知 | ❌→CV 线 | **项目成败所系，必须训练，但是 CV 不是 LLM 微调** |
| M02 焊缝/超标分割 | SAM 分割 | CV 感知 | ❌→CV 线 | 领域微调（CV） |
| M01 图像质量预检 | 密度/对比度/IQI | 经典 CV/度量 | ❌ | 阈值+传统算法，不训练 |
| M04 尺寸测量 | 像素标定+后处理 | 经典 CV/几何 | ❌ | 不训练 |
| M09 首件标定 | 质量评估+反馈环 | CV+控制 | ❌ | 基本不训练，**它还是 M00 的"验证兜底"** |
| M05 等级分级 | 缺陷+尺寸+材料→等级 | 可编码规则 | ❌→规则/KG | **明确不用 LLM**：可审计、零幻觉、适航要求 |
| M10 三路分料 | 规则+置信度阈值 | 规则 | ❌ | 不训练 |
| M06 标准匹配 | KG 精筛+向量重排 | 事实检索 | ❌→GraphForge | 本质 RAG，按边界划给 GraphForge |
| M07 报告生成 | 模板+受控生成 | 模板+轻 LLM | ❌ | 受控生成/槽填充，**Prompt 够，不微调** |
| M08 历史回溯 | 复用 M03 | CV 复用 | ❌ | 不另训 |
| **M00 工艺建模/程序生成** | 读文档/图纸→生成检测方案 | **拆 3 子任务** | ⚠️ **唯一候选** | 见下 |

**M00 再拆（唯一可能进平台的点）**：

| 子任务 | 瓶颈在 | 进平台? |
|--------|--------|:---:|
| A 文档抽取（工艺规程→规格/材质/焊缝数） | OCR/版面/术语，多数是上游问题 | 条件性 |
| B 几何坐标（图纸/STEP→焊缝坐标） | **有没有矢量源**——没有，微调也救不了 | ❌（解析器/数字化） |
| C 工艺推理（透照方式/扫描路径） | 可编码（按管径/壁厚查标准） | ❌（规则+LLM 提议规则校验） |

**结论**：该范例场景唯一可能用到本平台的是 **M00-A 文档抽取**，且触发条件极可能是**部署驱动**（涉密内网逼着用 GLM-4-9B 级小模型，微调/蒸馏把文档理解能力恢复到可用线），**而非领域驱动**。即——

> **该范例"用 LLM"是核心（M00 是客户头号需求：新工件→自动生成检测方案），但"需要微调"几乎不成立；
> 即便要微调，也是"部署驱动的能力恢复/蒸馏"，跟 HR/TCM 的"领域驱动"是两码事。**
> 这是"用 LLM ≠ 需要微调平台"最干净的一个实证。

## 6. 附录：探讨实录（推演链，便于追溯）

> 2026-06-13 的一次探讨，按问题推进顺序记录，保留推演链以备后续借鉴。以「航空 RT 质检范例」指代当时讨论的具体案例。

**Q1：该范例场景客观上需要语言模型/多模态微调吗，还是只需 CV 训练？**
- 逐模块定性：必须训练的格子全在 CV（M02 SAM + M03 YOLO），规则/经典 CV 模块不涉及学习。
- 三个 LLM/VLM 触点（M00 工艺理解 / M07 报告 / M06 标准匹配）都可靠 Prompt+受控生成+开箱 Embedding 上线。
- **结论：CV 是上线门槛（不训练系统不工作）；LLM/VLM 不微调系统照样跑。**

**Q2：M00 文本抽取都不微调，是不是代表垂直行业都不需要微调？**
- 否。决定变量不是"领域深浅"，是"领域智能装在哪"。该范例是极深垂直，但智能被外置到 KG/规则/CV，留给 LLM 的是搬运活。
- 引出 §1 第一性原理 + §3 两类垂直。

**Q3：把智能外置到 KG/规则/CV，是垂直行业最佳实践吗？**
- 是 A 类垂直（可编码+强监管+感知型）的最佳实践，有正式名字：**神经符号/混合 AI**；不是通用最佳实践。
- 求实补充：规则引擎是分级主力、KG 一半是配套一半是赶时髦；规则覆盖可编码大头 + 人工兜底隐性边缘。

**Q4：规则引擎能穷举领域所有规则吗？不能怎么办？**
- 不能，且永远不能（成文标准只编码显性部分 / 组合爆炸 / 新颖性 / 标准歧义）。
- 正解不是"穷举"，是**纵深防御**：规则覆盖大头 → **边界检测（知道自己不知道）** → 强制升级到人 → 保守失效（不确定不放行）→ **闭环回灌（人的结论固化成新规则/训练数据）**。完备性靠闭环渐近逼近，人工兜底让不完备始终安全。
- **关键：分级的"洞"甩给人，不甩给 LLM**（适航不许黑盒判定担责）。

**Q5：只用规则引擎，是不是就没法用 LLM 在垂直领域的泛化能力了？（客户核心需求是新工件→自动生成检测方案）**
- 前提纠正：架构不是"只用规则"，是混合——判定走规则，生成/理解走 LLM，两个不同模块。
- 客户核心需求（M00 新工件→方案）恰是泛化任务、LLM 主场，**完整保留，且应该用**。
- 关键一刀：**用 LLM ≠ 微调 LLM**。泛化是原生的，Prompt+架构就能调用，微调反而可能伤泛化。
- 为何 M00 可放手泛化、M05 不行：M00 输出是**可验证可逆的草案**（首件标定试拍+仿真+工艺师审核兜底），M05 输出是**最终不可逆的判定**（工件直接出厂，无安全兜底）。

**Q6：M00"用 LLM 原生泛化+不微调"够不够支撑客户核心诉求？什么情况下顶不住、不得不微调？**
- M00 拆 3 子任务（A 文档抽取 / B 几何坐标 / C 工艺推理），可靠性瓶颈 80% 不在"LLM 够不够聪明"，在数据可得性与架构选型。
- 真正的可靠性标尺不是"完美"，是"给定兜底后省力率够不够"——**"顶不住" = 工艺师返工率高到没省下人力，即 ROI 垮掉**，不是"偶尔出错"。
- 真正逼微调的只有两条：①**部署驱动**（涉密小模型能力恢复/蒸馏，最可能命中本范例）②**领域驱动**（系统性术语/格式误读，且便宜修法已尽 + 有数据 + ROI 撑得起）。
- 反例（绝不微调，往上游修）：瓶颈是 OCR/版面、缺矢量源、可编码、仅偶尔错且有兜底。

→ 以上 6 问收敛成本文 §1–§5 的判据框架。
