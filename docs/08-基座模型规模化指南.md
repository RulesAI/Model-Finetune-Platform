> **创建**：2026-05-25
> **用途**：跨规模基座模型（1B-7B / 9B-14B / 22B-32B / 70B / MoE）的工具集配置矩阵

## 一、为什么需要本文档

不同规模基座模型的训练 / 推理需求差异 **10-100×**。工具集（ms-swift / LLaMA-Factory / LMDeploy）虽然 600+ 模型通用，但**配置必须场景化**：

| 模型规模 | 9B (HR 当前) | 32B (HR 升级) | 70B | MoE 355B |
|---------|------------|------------|-----|----------|
| 训练 GPU 数量 | 1 卡 | **4 卡** | 8 卡 | **16-32 卡多机** |
| 训练显存 | A100 80G × 1 | A100 80G × 4 | A100 80G × 8 | H100 80G × 16+ |
| 学习率 | 2e-4 | **1e-4** | 5e-5 | 2e-5 |
| DeepSpeed 配置 | 无 | **ZeRO-2** | ZeRO-2 | ZeRO-3 + 多机 |
| Unsloth backend | ✅ | ❌ | ❌ | ❌ |
| 量化推理显存 | 5 GB | 16-18 GB | 40 GB | 200 GB（FP16，无量化）|

**本文档统一回答**：每档配置应该怎么改？v4 工具集是否够用？

## 四、F7 校正：DeepSpeed ZeRO 配置决策树

### 4.1 关键技术发现

业界实证：**LoRA + DeepSpeed ZeRO-3 不兼容** —— ZeRO-3 partition 模型参数到各 GPU，破坏 LoRA 适配器的梯度流。

> Source: "DeepSpeed ZeRO-3 fails with LoRA on this architecture because ZeRO-3 partitions model parameters across GPUs, which **breaks gradient flow when LoRA adapters are added**. The solution is to use **ZeRO-2**."

### 4.2 ZeRO 阶段对比

| ZeRO 阶段 | 分片对象 | LoRA 兼容性 | 显存节省 | 适用场景 |
|----------|---------|------------|---------|---------|
| **ZeRO-0** | 不分片（DDP）| ✅ | 0 | 单卡或小模型 |
| **ZeRO-1** | 优化器状态 | ✅ | 4× | 中等模型 |
| **ZeRO-2** ⭐ | 优化器状态 + 梯度 | ✅ **LoRA 推荐** | 8× | **LoRA + 多卡首选** |
| ZeRO-3 | + 模型参数 | ❌ **破坏 LoRA** | 16× | **仅全参 SFT / CPT** |

### 4.3 选择决策树

```
训练方式 + 模型规模 → DeepSpeed 配置
│
├── LoRA / QLoRA（任何规模）
│   ├── 单卡 → 不用 DeepSpeed
│   ├── 多卡（2-8 卡）→ **ZeRO-2** ⭐
│   └── 多机 → ZeRO-2 + 多机配置
│
├── 全参 SFT
│   ├── 9B 单卡 → ZeRO-2
│   ├── 32B 多卡 → ZeRO-2 或 ZeRO-3
│   └── 70B+ → ZeRO-3 必需
│
└── CPT（继续预训练）
    └── 任何规模 → ZeRO-3（数据量大，需最大显存节省）
```

### 4.4 ms-swift 配置示例

```yaml
# configs/ms_swift/scenes/<scene>/sft-32b.yaml
model_id_or_path: ZhipuAI/glm-4-32b-0414
sft_type: lora
deepspeed: zero2          # ⭐ 32B + LoRA 必须 ZeRO-2
learning_rate: 1e-4       # ⭐ 32B 比 9B 降一档
per_device_train_batch_size: 4
gradient_accumulation_steps: 8
num_train_epochs: 3

target_modules:
  - q_proj
  - k_proj
  - v_proj
  - o_proj
  - gate_proj
  - up_proj
  - down_proj
lora_rank: 16
lora_alpha: 32
```

## 五、F8 校正：规模化超参矩阵详解

### 5.1 学习率规模化规则

业界经验法则：参数量翻倍，学习率约减半。

```
模型规模    学习率
7B          3e-4
9B-14B      2e-4   ← HR 主推
32B         1e-4   ← ⭐ 比 9B 降一档
70B         5e-5
MoE 355B    2e-5
```

**判定依据**：大模型已经"知道"更多，微调时小步走更稳，否则灾难性遗忘。

### 5.2 batch size 规模化规则

**核心约束**：effective batch size（per_device × num_gpus × grad_accum）**保持约 32**。

| 档位 | per_device | num_gpus | grad_accum | effective |
|------|-----------|---------|------------|-----------|
| S1 (7B) | 16 | 1 | 2 | 32 |
| S2 (9B) | 8 | 1 | 4 | 32 |
| S3 (32B) | 4 | 4 | 2 | 32 |
| S4 (70B) | 2 | 8 | 2 | 32 |
| S5 (MoE) | 1 | 16 | 2 | 32 |

per_device batch 必须根据显存调小，但 effective batch 不变保证收敛速度。

### 5.3 LoRA rank 选择

| 档位 | r | alpha | 注释 |
|------|---|-------|------|
| S1-S2 | 16 | 32 | 业界主流 |
| S3 | 16 | 32 | 不需要 r=32（边际收益小）|
| S4 | 16-32 | 32-64 | 大模型可放大 |
| S5 (MoE) | 32 | 64 | MoE 路由学习需更高表达力 |

**别盲目放大 r**：r=8→16 收益显著，r=16→32 收益边际，r=32→64 多数情况浪费。

### 5.4 target_modules 选择

| 档位 | target_modules | 注释 |
|------|--------------|------|
| 保守 | `q_proj,v_proj` | 最小化训练参数 |
| **平衡** ⭐ | `q_proj,k_proj,v_proj,o_proj` | 通用最佳 |
| 激进 | `all-linear`（含 MLP）| 最大效果，但参数多 4× |

HR 9B / 32B 都推荐**平衡方案**：4 个 attention proj。

## 十、典型升级路径

### 10.1 HR 场景演进示例

```
W0 启动：S2 (GLM-4-9B-Chat-1M)
  ↓ 5 周训练完
W5：评估结果

├── 业务方满意 → 维持 S2 上线（推荐路径）
│
├── 业务方要求更强能力（多步推理 / 复杂方案生成）
│   └── 升级 S3 (GLM-4-32B-0414)
│       ↓ 6 周重训
│       └── 评估 vs S2 提升度
│           ├── 提升 ≥ 15% → 切 S3
│           └── 提升 < 15% → 退回 S2（不值得多花 4× 成本）
│
└── 业务方要求 Agent 工具调用
    └── 不升级基座，下游业务系统加 Tool use 编排（GraphForge / EvoChain）
```

### 10.2 跨场景升级决策矩阵

| 客户类型 | 推荐起步档 | 升级触发条件 | 升级目标档 |
|---------|----------|-----------|----------|
| 中小企业 POC | S1 | 业务方接受度高 | S2 |
| **HR 国企** ⭐ | **S2** | 复杂业务问题 SFT 不收敛 | S3 |
| 制造企业 | S2 | 多模态需求 | S2 多模态 |
| 大型央企 | S2 → S3 | 多场景复用 | S3 / S4 |
| 互联网大客户 | S3 | Agent 复杂度 | S4 / S5 |

## 十一、风险与避坑清单

### 11.1 通用风险

| 风险 | 规避策略 |
|------|---------|
| 盲目追求大模型 | 先 S2 跑通业务，**有明确收益证据**再升级 |
| 学习率不降导致 32B 灾难性遗忘 | 严格按 §五.1 学习率表 |
| DeepSpeed ZeRO-3 + LoRA 报错 | **改 ZeRO-2** ⭐ F7 |
| Unsloth 在 32B 多卡崩溃 | **禁用 Unsloth** + 用 ms-swift 原生 backend |
| 单卡显存不够 OOM | 降 batch / 启用 gradient checkpointing / 切多卡 |
| effective batch 不一致影响收敛 | 严格按 §五.2 batch 矩阵 |
| QLoRA 4bit 量化训练效果掉点 | 仅 S4+ 资源紧时用，S1-S3 用 LoRA |

### 11.2 不推荐场景

| 不推荐 | 理由 |
|--------|------|
| ❌ HR 直接上 S4 (70B) | 业务收益 < 成本 4×，先 S2/S3 验证 |
| ❌ S5 MoE 作 v1 首选 | 当前工具集支撑度仅 70%，配置坑多 |
| ❌ S1 (7B) 涉密内网 | 业务方对小模型说服力低，不如 S2 |
| ❌ 同时跑 S2 + S3 双基座对比 5 周 | 算力翻倍，建议串行（先 S2 5 周 → 满意维持 / 不满意切 S3 6 周）|

### 11.3 信号判定（什么时候真该升级）

| 信号 | 是否升级 |
|------|---------|
| S2 业务测试集人工评估 ≥ 4.0/5 | ✅ 不需要升级，已达标 |
| S2 评估 ≥ 4.5/5 + 业务方提出更高需求 | ⚠️ 考虑升级 S3 |
| S2 业务测试集 < 3.5/5 + 训练数据 / 超参充分调优后仍不行 | ✅ 真该升级 S3 |
| 仅"想看看 32B 效果好不好" | ❌ 不要 |

## 十三、一句话摘要

**基座模型规模化的核心是 5 档（S1 1B-7B / S2 9B-14B / S3 22B-32B / S4 70B+ / S5 MoE 355B）+ 2 个校正点（F7：LoRA 必配 DeepSpeed ZeRO-2 不能 ZeRO-3 / F8：参数规模翻倍学习率约减半），v4 工具集对 S1-S3 支撑度 ≥ 96%（HR 9B→32B 升级仅改 YAML 不换工具），S4 需 QLoRA + 多机扩展，S5 MoE 需 v5+ 重大扩展。**
